Fase 0 — Setup iniziale

Da zero al remoto

Come portare un progetto locale su Azure DevOps (o GitHub) partendo da git init, oppure clonarne uno già esistente.

0.1 — Inizializzare il repo e fare il primo commit

Il punto di partenza assoluto: una cartella vuota (o con dei file già presenti) che vuoi mettere sotto controllo versione.

# Entra nella cartella del progetto
cd C:\progetti\mio-progetto

# Inizializza il repo Git locale
git init

# Rinomina il branch di default (main è la convenzione moderna)
git branch -M main

# Aggiungi tutti i file e fai il primo commit
git add .
git commit -m "chore: primo commit"
Perché git branch -M main?

Git crea storicamente il branch di default come master. Azure DevOps e GitHub usano main come convenzione attuale. Il flag -M forza il rename anche se esiste già un branch con quel nome. Farlo subito evita disallineamenti col remoto.

Senza almeno un commit, il branch non esiste

Git non crea fisicamente il branch main finché non c'è almeno un commit. Se provi a pushare su un repo vuoto locale, non trovi nulla da mandare. Commit prima, push dopo.

0.2 — Collegare il remoto

Prima di questo step, devi aver creato il repo su Azure DevOps (o GitHub) dalla UI — vuoto, senza README né .gitignore. Se il remoto ha già dei commit, il push iniziale andrà in conflitto.

# Azure DevOps
git remote add origin https://<organizzazione>.visualstudio.com/<Progetto>/_git/<NomeRepo>

# Oppure GitHub
git remote add origin https://github.com/<username>/<nome-repo>.git
ParteSignificato
git remote addRegistra un nuovo remote nel repo locale
originAlias del remote — convenzione universale, potresti chiamarlo diversamente
https://...URL del repo remoto — lo trovi nella UI di Azure DevOps / GitHub con il tasto "Clone"

Per verificare che il remote sia stato aggiunto correttamente:

git remote -v

Dovresti vedere origin listato sia per fetch che per push.

0.3 — Push iniziale

git push -u origin main
ParteSignificato
pushManda i commit locali al remoto
-uAbbreviazione di --set-upstream: collega il branch locale main al remoto origin/main
originIl remote di destinazione
mainIl branch da pushare
Il -u vale solo la prima volta

Dopo il primo git push -u origin main, Git ricorda il collegamento. Da quel momento basta git push senza specificare remote e branch.

Sequenza completa in un colpo

git init
git branch -M main
git add .
git commit -m "chore: primo commit"
git remote add origin https://<organizzazione>.visualstudio.com/<Progetto>/_git/<NomeRepo>
git push -u origin main

0.4 — Alternativa: git clone

Se il repo esiste già sul remoto e vuoi portarlo in locale, non hai bisogno di git init né di git remote add. Un solo comando fa tutto:

git clone https://<organizzazione>.visualstudio.com/<Progetto>/_git/<NomeRepo>

Questo crea una cartella col nome del repo, scarica tutto il contenuto e imposta automaticamente origin puntando al remoto. Sei operativo immediatamente.

ScenarioApproccio corretto
Progetto nuovo, parte dal tuo PCgit init → commit → remote add → push -u
Repo già esistente su Azure/GitHubgit clone <url> — tutto automatico
Azure DevOps: crea il repo vuoto dalla UI prima

A differenza di GitHub che in certi flussi crea il repo al volo, su Azure DevOps devi sempre creare il repo dalla UI prima di poter fare il push. Vai su Repos → New repository, scegli Git, lascia vuoto (niente README), poi torna al terminale.

0.5 — Annullare un git add

Hai fatto git add per sbaglio e vuoi togliere i file dalla staging area? Dipende da dove sei nel flusso.

Caso A — Hai già almeno un commit

# Tutto fuori dalla staging
git restore --staged .

# Solo un file
git restore --staged nomefile.txt

Le modifiche rimangono sul disco — vengono solo tolte dalla staging area. Nessun danno.

Caso B — Non hai ancora nessun commit (HEAD non esiste)

Se usi git restore --staged prima del primo commit, Git risponde con:

fatal: could not resolve 'HEAD'

È normale: HEAD non esiste ancora perché non c'è nessun commit. Il comando da usare in questo caso è:

# Tutto fuori dalla staging
git rm --cached .

# Solo un file
git rm --cached nomefile.txt
Perché --cached?

Il flag --cached dice a Git di operare solo sulla staging area, senza toccare i file sul disco. Senza quel flag, git rm eliminerebbe anche il file fisicamente.

SituazioneComando
Dopo almeno un commitgit restore --staged .
Prima del primo commitgit rm --cached .
Vuoi anche buttare le modifiche dal discogit restore . ⚠️ irreversibile
Fase 1 — Modulo 2 del tutorial

Branching & Merge

Il punto di partenza. Fast-forward vs merge esplicito, e come leggere il grafo risultante.

1.1 — Fast-forward merge

Crea feature/login da dev, fai 2 commit, torna su dev e merga senza --no-ff.

git switch dev
git switch -c feature/login
# modifica o crea un file, poi:
git add .
git commit -m "feat: aggiunta pagina login"
# modifica ancora, poi:
git add .
git commit -m "feat: aggiunta validazione form login"
git switch dev
git merge feature/login
git log --oneline --graph --all

Log reale ottenuto

* fb21692 (HEAD -> dev, feature/login) feat: aggiunta validazione pagina login * 7eeeb37 feat: aggiunta pagina login * c387ac1 (origin/main, origin/dev, main) primo commit
Riga per riga:
* fb21692 (HEAD -> dev, feature/login) — il commit più recente. HEAD -> dev = sei su dev. feature/login punta allo stesso commit. Due puntatori, un solo commit — questa è la fast-forward: Git ha solo spostato il puntatore di dev in avanti, senza creare un commit di merge.

* 7eeeb37 — il commit precedente di feature/login. Nessun ref accanto = commit intermedio, non punta nessun branch.

* c387ac1 (origin/main, origin/dev, main) — il primo commit del repo. origin/main e origin/dev sono ancora qui = non hai ancora pushato i nuovi commit. main = il branch main locale, fermo al primo commit.
Concetto chiave

La fast-forward è "invisibile" nella storia: non lascia traccia che quei commit venissero da un branch separato. Per questo nel team si usa --no-ff.

1.2 — Merge con --no-ff

Stessa cosa ma forzando un commit di merge esplicito.

git switch -c feature/dashboard
# crea dashboard.txt, poi:
git add .
git commit -m "feat: aggiunta pagina dashboard"
# modifica dashboard.txt aggiungendo i widget, poi:
git add .
git commit -m "feat: aggiunta widget statistiche"
git switch dev
git merge --no-ff feature/dashboard -m "merge: feature/dashboard in dev"
git log --oneline --graph --all

Log reale ottenuto

* 08503e7 (HEAD -> dev) merge: feature/dashobard in dev |\ | * eff6980 (feature/dashboard) feat: aggiunta widget statistiche | * 35bf56c feat: aggiunta pagina dashboard |/ * fb21692 (feature/login) feat: aggiunta validazione pagina login * 7eeeb37 feat: aggiunta pagina login * c387ac1 (origin/main, origin/dev, main) primo commit
Riga per riga:
*   08503e7 (HEAD -> dev) — il commit di merge. *   con tre spazi indica un merge commit (ha due genitori). HEAD -> dev = sei qui ora.

|\ — i due branch si riuniscono qui. Da questo punto in giù la storia si divide in due binari paralleli.

| * eff6980 (feature/dashboard) — commit sul binario di destra, cioè feature/dashboard. Il | * significa "commit su un branch parallelo".

| * 35bf56c — secondo commit sempre su feature/dashboard, ancora sul binario di destra.

|/ — i due binari si biforcano (verso il basso = verso il passato). È qui che è stato creato feature/dashboard.

* fb21692 (feature/login) — il punto di partenza comune. Da qui i due branch si sono separati. Nota: feature/login punta allo stesso commit di dev perché era una fast-forward.

* c387ac1 (origin/main, origin/dev, main) — il primo commit. I ref remoti sono ancora qui = nessun push fatto finora.
Tip

Per vedere solo i commit esclusivi di un branch rispetto a un altro (utile prima del merge): git log --oneline dev..feature/dashboard. Dopo il merge non mostra nulla — quei commit fanno già parte di dev.

1.3 — Come leggere il grafo

Il grafo si legge dall'alto verso il basso — in cima c'è il presente, in fondo il passato.

SimboloSignificato
*Un commit
|\I branch si riuniscono (commit di merge)
|/Il branch si biforca (punto di creazione)
| *Un commit su un branch parallelo
HEAD -> devDove sei posizionato ora (dev locale)
origin/devDove è il remoto — se diverso da dev, sono divergiti
Prima di passare alla Fase 2 — pusha!

Allinea il remoto con quello che hai fatto in locale:

git push origin dev

Così origin/dev torna allineato a dev e il log nella prossima fase sarà coerente.

Fase 2 — Modulo 5 del tutorial

Conflitti intenzionali

Il modo migliore per imparare a gestire i conflitti è crearli apposta.

2.1 — Conflitto manuale

Modifica la stessa riga dello stesso file su due branch diversi, poi tenta il merge.

git switch dev
git switch -c feature/conflitto-test
# Apri dashboard.txt, modifica la prima riga:
# "Dashboard v2 — modificato da feature/conflitto-test"
git add .
git commit -m "feat: modifica dashboard da feature/conflitto-test"
git switch dev
# Apri dashboard.txt, modifica la STESSA riga:
# "Dashboard v2 — modificato da dev"
git add .
git commit -m "feat: modifica dashboard da dev"
git merge --no-ff feature/conflitto-test

Output del merge in conflitto

Auto-merging api/src/main/java/.../dashboard.txt
CONFLICT (content): Merge conflict in dashboard.txt
Automatic merge failed; fix conflicts and then commit the result.

Cosa trovi nel file

<<<<<<< HEAD
Dashboard v2 — modificato da dev
=======
Dashboard v2 — modificato da feature/conflitto-test
>>>>>>> feature/conflitto-test

Modifica il file a mano, rimuovi tutti i marcatori, salva, poi:

git add .
git commit -m "merge: risolto conflitto dashboard"
git log --oneline --graph --all

Log reale ottenuto

* f19fffe (HEAD -> dev) merge: risolto conflitto dashboard |\ | * 2690e79 (feature/conflitto-test) feat: modifica dashboard da feature/conflitto-test * | 8e721b4 feat: modifica dashboard da dev |/ * 21f79e7 merge: feature/dashboard in dev |\ | * 002101c (feature/dashboard) feat: aggiunta widget statistiche | * 6618858 feat: aggiunta pagina dashboard |/ * 1db4b2a (feature/login) feat: aggiunta validazione pagina login
Riga per riga:
*   f19fffe (HEAD -> dev) — il commit di merge che ha risolto il conflitto. I tre spazi dopo * indicano merge commit. HEAD -> dev = sei qui ora.

|\ — da qui la storia si divide in due binari paralleli. Leggi: i due branch si riuniscono in questo merge.

| * 2690e79 (feature/conflitto-test) — commit sul binario di destra (feature branch). È la modifica che aveva fatto il feature branch sulla stessa riga.

* | 8e721b4 — commit sul binario di sinistra (dev). È la modifica fatta direttamente su dev. Nota: * | invece di | * — siamo sul ramo principale.

|/ — i due binari si biforcano verso il passato. È qui che è stato creato feature/conflitto-test da dev.

Questo schema a diamante — |\ poi * |/| * poi |/ — è la firma visiva di qualsiasi merge con conflitto (o senza). Più lo vedi, più diventa automatico riconoscerlo.

2.2 — Risoluzione automatica con -X ours

Crea un conflitto identico ma usa -X ours per risolverlo senza toccare i file.

git switch -c feature/conflitto-ours
# Modifica dashboard.txt: "Dashboard v3 — versione dal feature branch"
git add .
git commit -m "feat: dashboard v3 da feature branch"
git switch dev
# Modifica la STESSA riga: "Dashboard v3 — versione da dev"
git add .
git commit -m "feat: dashboard v3 da dev"
git merge --no-ff feature/conflitto-ours -X ours -m "merge: conflitto risolto con ours"

Apri dashboard.txt — contiene: "Dashboard v3 — versione da dev". Ha vinto il branch corrente.

2.3 — Risoluzione automatica con -X theirs

git switch -c feature/conflitto-theirs
# Modifica dashboard.txt: "Dashboard v4 — versione dal feature branch"
git add .
git commit -m "feat: dashboard v4 da feature branch"
git switch dev
# Modifica la STESSA riga: "Dashboard v4 — versione da dev"
git add .
git commit -m "feat: dashboard v4 da dev"
git merge --no-ff feature/conflitto-theirs -X theirs -m "merge: conflitto risolto con theirs"

Apri dashboard.txt — contiene: "Dashboard v4 — versione dal feature branch". Ha vinto il branch mergato.

FlagChi vince
-X oursIl branch su cui sei (quello che riceve il merge)
-X theirsIl branch che stai mergando
Attenzione

Se usi rebase invece di merge, ours e theirs si invertono. Questo è approfondito nel Modulo 6 del tutorial.

Prima di passare alla Fase 3 — pusha!

Hai mergato in locale — allinea il remoto:

git push origin dev
Fase 3 — Moduli 3 & 4 del tutorial

Remote & PR su Azure DevOps

Simula il workflow reale da team: pusha un feature branch, apri una PR, completa il merge e riallinea il locale.

3.1 — Push branch e apertura PR

git switch dev
git switch -c feature/navbar
# crea/modifica un file, poi:
git add .
git commit -m "feat: aggiunta navbar"
git add .
git commit -m "feat: aggiunta link navbar"
git push origin feature/navbar

Su Azure DevOps apparirà il banner "You updated feature/navbar — Create a pull request". Clicca, imposta dev come branch di destinazione (non master!), e clicca Create. Poi clicca Complete per finalizzare il merge.

Attenzione

La PR non fa il merge da sola: devi cliccare Complete per eseguirlo. Azure DevOps usa sempre --no-ff di default.

3.2 — Fetch e allineamento

Dopo aver completato la PR su Azure DevOps, il merge è avvenuto sul remoto ma il tuo locale non lo sa ancora. Prima di tutto torna su dev, poi scarica lo stato del remoto:

git switch dev
git fetch origin
git log --oneline --graph --all

Log dopo fetch — prima del pull

* 9697c2c (origin/dev) Merged PR 2: feat: aggiunta navbar |\ | * d1094a0 (HEAD -> feature/navbar, origin/feature/navbar) feat. aggiunta link navbar | * 3a7a3e1 feat. aggiunta navbar |/ * 31b5e56 (dev) merge: conflitto risolto con theirs |\ | * 829d4f6 (feature/conflitto-theirs) feat: dashboard v4 da feature branch * | 6eb5767 feat dashboard v4 da dev |/ * c62c7ce merge: conflitto risolto con ours
Riga per riga:
*   9697c2c (origin/dev) — il commit di merge creato dalla PR su Azure DevOps. Nota: solo origin/dev, non dev. Il remoto è avanzato ma il tuo locale non sa ancora nulla.

|\ — il merge della PR ha unito feature/navbar in dev sul remoto.

| * d1094a0 (HEAD -> feature/navbar, origin/feature/navbar) — sei ancora posizionato su feature/navbar (HEAD ->). Il branch esiste sia in locale che sul remoto, entrambi allo stesso commit.

| * 3a7a3e1 — il primo commit di feature/navbar.

|/ — il punto di biforcazione di feature/navbar da dev.

*   31b5e56 (dev) — qui sta dev locale. È rimasto indietro rispetto a origin/dev che è già a 9697c2c. Questo è il segnale che devi fare git pull. 
git pull origin dev
git log --oneline --graph --all

Log dopo il pull — allineato

* 9697c2c (HEAD -> dev, origin/dev) Merged PR 2: feat: aggiunta navbar |\ | * d1094a0 (origin/feature/navbar, feature/navbar) feat. aggiunta link navbar | * 3a7a3e1 feat. aggiunta navbar |/ * 31b5e56 merge: conflitto risolto con theirs
Riga per riga:
*   9697c2c (HEAD -> dev, origin/dev) — confronta con il log precedente (post-fetch): prima c'era solo (origin/dev) qui, mentre dev locale era fermo a 31b5e56. Il pull ha scaricato i commit dal remoto e spostato il puntatore locale in avanti. Ora dev e origin/dev puntano allo stesso commit — locale e remoto allineati.

| * d1094a0 (origin/feature/navbar, feature/navbar) — feature/navbar non ha HEAD -> perché sei passato su dev. Il branch esiste ancora in locale e sul remoto, ma non sei posizionato lì.

3.3 — git pull --rebase

Hai già visto git pull origin dev che fa un merge implicito. La variante --rebase fa la stessa cosa ma invece di creare un merge commit, riapplica i tuoi commit locali sopra quelli scaricati dal remoto, mantenendo la history lineare.

# Simula il caso: sei su dev, hai un commit locale non pushato
# e nel frattempo origin/dev è avanzato
git switch dev
# crea pullrebase.txt, poi:
git add .
git commit -m "feat: commit locale non pushato"

# ora allinea con il remoto usando --rebase invece di merge
git pull --rebase origin dev
ParteSignificato
pullScarica i commit dal remoto e li integra in locale
--rebaseInvece di creare un merge commit, riapplica i tuoi commit locali sopra quelli appena scaricati
origin devIl remote e il branch da cui scaricare

Confronto visivo: pull normale vs pull --rebase

Supponiamo che origin/dev abbia avanzato di un commit (X) e tu in locale hai un commit (Y) non ancora pushato:

OperazioneRisultato nel log
git pull origin devCrea un merge commit M con due genitori (X e Y) — la history si biforca e si ricongiunge. Cherry-pick successivi diventano più complessi.
git pull --rebase origin devSposta Y sopra X — la history rimane lineare, come se avessi committato Y dopo X. Cherry-pick rimane semplice e chirurgico.
Confronto con git rebase origin/dev

git pull --rebase origin dev è equivalente a:

git fetch origin
git rebase origin/dev

La differenza è solo pratica: pull --rebase fa entrambi i passi in uno. Il risultato finale è identico.

E -X ours / -X theirs?

Le strategie -X ours e -X theirs che hai visto nella Fase 2 si applicano solo al merge, non al rebase. Con pull --rebase non puoi usarle direttamente — in caso di conflitto durante il rebase devi risolverlo manualmente file per file e poi:

git add .
git rebase --continue

Oppure, se vuoi abortire e tornare allo stato precedente:

git rebase --abort
Attenzione: ours e theirs si invertono nel rebase

Durante un rebase, i ruoli si invertono rispetto al merge: ours = i commit del branch su cui stai rebasando (cioè origin/dev), theirs = i tuoi commit locali che vengono riapplicati. L'opposto di quanto ti aspetteresti. Questo è approfondito nel Modulo 6 del tutorial principale.

Log reale ottenuto

Prima del pull --rebase — il disallineamento:

* 45e4887 (HEAD -> dev) prova per git pull --rebase * 7851a8c (origin/dev) Updated login.txt per git pull --rebase
45e4887 (HEAD -> dev) — il commit locale non pushato. HEAD -> dev = sei qui, ma origin/dev non c'è accanto = il remoto è fermo a un commit diverso.

7851a8c (origin/dev) — il commit fatto direttamente dalla UI di Azure DevOps, che simula il push di un collega. I due branch sono divergiti: ognuno ha un commit che l'altro non ha.

Dopo git pull --rebase origin dev:

* 45e4887 (HEAD -> dev) prova per git pull --rebase * 7851a8c (origin/dev) Updated login.txt per git pull --rebase
History lineare — nessuna biforcazione, nessun merge commit. Il commit locale (45e4887) è stato riapplicato sopra quello di origin (7851a8c) come se fosse sempre stato lì dopo.

origin/dev è fermo a 7851a8c = non hai ancora pushato. Quando farai git push origin dev, i due puntatori si allineeranno sullo stesso commit.

Confronta con come sarebbe apparso con un git pull normale:

* xxxxxxx (HEAD -> dev) Merge branch 'dev' of https://... |\ | * 7851a8c (origin/dev) Updated login.txt per git pull --rebase * | 45e4887 prova per git pull --rebase |/
Quella biforcazione non c'è con --rebase. Un merge commit in più, due genitori, history non lineare — tutto questo complica i cherry-pick successivi.

Quando usare pull --rebase invece di pull normale

SituazionePreferisci
Vuoi history lineare, cherry-pick semplicigit pull --rebase — il tuo caso tipico nel workflow del team
Hai solo commit locali non ancora pushatigit pull --rebase — situazione ideale, nessun rischio
Vuoi tracciare esplicitamente quando hai integrato il remotogit pull — il merge commit documenta il momento dell'integrazione
Hai commit già pushati e condivisi con altrigit pull — il rebase riscrive gli hash, pericoloso su branch condivisi
Regola pratica per il tuo team

Nel vostro workflow con cherry-pick verso integration, la history lineare è una priorità — git pull --rebase dovrebbe essere la scelta di default quando aggiorni un feature branch o dev. Usa il pull normale solo quando devi tracciare esplicitamente un'integrazione o quando il branch è già condiviso e altri ci hanno già lavorato sopra.

Fase 4 — Modulo 6 del tutorial

Rebase

Linearizzare la storia e pulire i commit prima di una PR.

Regola d'oro

Mai fare rebase su branch condivisi (dev, master). Solo su branch tuoi, non ancora pushati o su cui lavori solo tu.

4.1 — Rebase base

Simula il caso reale: mentre lavori sul tuo branch, qualcuno ha pushato su dev.

git switch dev
git switch -c feature/rebase-test
# crea rebase1.txt, poi:
git add .
git commit -m "feat: aggiunta file rebase 1"
# crea rebase2.txt, poi:
git add .
git commit -m "feat: aggiunta file rebase 2"
git switch dev
# modifica un file diverso, poi:
git add .
git commit -m "feat: commit su dev nel frattempo"

Log prima del rebase — branch divergiti

* b3aa22e (HEAD -> dev) feat: commit su dev nel frattempo | * bcf2c4e (feature/rebase-test) feat: aggiunta file rebase 2 | * 8fbaef0 feat: aggiunta file rebase 1 |/ * 9697c2c (origin/dev) Merged PR 2: feat: aggiunta navbar
Riga per riga:
* b3aa22e (HEAD -> dev) — l'ultimo commit su dev ("commit su dev nel frattempo"). HEAD -> dev = sei su dev. Questo commit non esiste sul binario di feature/rebase-test.

| * bcf2c4e (feature/rebase-test) — il commit più recente di feature/rebase-test. Sta su un binario parallelo (| *) — non discende da b3aa22e.

| * 8fbaef0 — il primo commit di feature/rebase-test. Anch'esso sul binario parallelo.

|/ — punto di biforcazione. Da qui i due branch hanno preso strade diverse. L'antenato comune è il commit sotto: 9697c2c.

*   9697c2c (origin/dev) — l'antenato comune. Sia dev che feature/rebase-test sono partiti da qui. Il rebase sposterà i commit del feature branch "sopra" b3aa22e, eliminando questa biforcazione. 
git switch feature/rebase-test
git rebase dev
git log --oneline --graph --all

Log dopo il rebase — storia lineare

* 1721453 (HEAD -> feature/rebase-test) feat: aggiunta file rebase 2 * dd361f6 feat: aggiunta file rebase 1 * b3aa22e (dev) feat: commit su dev nel frattempo * 9697c2c (origin/dev) Merged PR 2: feat: aggiunta navbar
Riga per riga:
* 1721453 (HEAD -> feature/rebase-test) — il secondo commit del feature branch, riscritto. Contenuto identico a bcf2c4e di prima, ma hash diverso perché ora il suo genitore è b3aa22e invece di 9697c2c.

* dd361f6 — il primo commit del feature branch, anch'esso riscritto (8fbaef0 → dd361f6). Stesso contenuto, nuovo genitore.

* b3aa22e (dev) — il commit "nel frattempo" su dev. Invariato — stesso hash di prima. Il rebase non tocca dev.

*   9697c2c (origin/dev) — l'antenato comune. Invariato.

La linea dritta è la prova del rebase riuscito: non c'è più la biforcazione | */|/. I commit sembrano essere stati fatti direttamente su dev, in sequenza. Ma attenzione: gli hash vecchi (bcf2c4e, 8fbaef0) esistono ancora nel reflog per 90 giorni.

4.2 — Interactive rebase (squash)

Unisci più commit in uno solo prima di pushare — utile per pulire una serie di commit "sporchi".

# aggiungi un terzo commit a feature/rebase-test
# crea rebase3.txt, poi:
git add .
git commit -m "feat: aggiunta file rebase 3"
git rebase -i HEAD~3
ParteSignificato
rebaseRiscrivi la storia del branch spostando o modificando commit
-iModalità interattiva — si apre un editor dove scegli cosa fare con ogni commit
HEAD~3Lavora sugli ultimi 3 commit a partire da HEAD. ~3 significa "3 commit indietro"

Si apre l'editor. Modifica pick in s (squash) per i commit 2 e 3:

pick dd361f6 feat: aggiunta file rebase 1
s 1721453 feat: aggiunta file rebase 2
s b2008f4 feat: aggiunta file rebase 3

Salva e chiudi. Si apre un secondo editor per il messaggio finale — scrivi:

feat: aggiunta file rebase completo

Log dopo lo squash

* f9413bd (HEAD -> feature/rebase-test) feat: aggiunta file rebase completo | * 1721453 (origin/feature/rebase-test) feat: aggiunta file rebase 2 | * dd361f6 feat: aggiunta file rebase 1 |/ * b3aa22e (origin/dev, dev) feat: commit su dev nel frattempo
Riga per riga:
* f9413bd (HEAD -> feature/rebase-test) — il commit unico risultato dello squash. Contiene le modifiche di tutti e 3 i commit originali.

| * 1721453 (origin/feature/rebase-test) — il remoto è ancora al vecchio stato con 2 commit separati. | * = binario parallelo. Locale e remoto sono divergiti.

| * dd361f6 — il secondo vecchio commit sul remoto.

|/ — il punto di divergenza: da qui la storia locale e quella remota sono diverse.

* b3aa22e (origin/dev, dev) — l'antenato comune. Sia locale che remoto concordano su questo punto.

Questa biforcazione tra locale e remoto è esattamente perché non puoi fare git push normale — il remoto ha una storia diversa. Serve --force-with-lease per sovrascriverla.

4.3 — --force-with-lease

git push --force-with-lease origin feature/rebase-test

Output del push forzato

+ 1721453...f9413bd feature/rebase-test -> feature/rebase-test (forced update)
Cosa osservare: il + e forced update confermano che Git ha sovrascritto la storia remota. 1721453...f9413bd mostra il vecchio hash sostituito dal nuovo.
Perché --force-with-lease e non --force?

--force sovrascrive il remoto incondizionatamente. --force-with-lease sovrascrive solo se nessun altro ha pushato nel frattempo — se qualcuno ha aggiunto commit dopo il tuo ultimo fetch, il push viene rifiutato e non perdi il loro lavoro.

Cosa succede se qualcuno ha pushato nel frattempo

In questo caso il push viene rifiutato invece di sovrascrivere alla cieca — è proprio questo lo scopo di --force-with-lease:

! [rejected]        feature/rebase-test -> feature/rebase-test (stale info)
error: failed to push some refs to '...'
Ma se il branch è "mio", quando succede davvero?

Se il branch è esclusivamente tuo e lo usi da una sola macchina, questo scenario non dovrebbe verificarsi mai. Capita in casi come:

  • Lavori sullo stesso branch da due macchine (laptop + PC fisso) e hai pushato da una dimenticandotene
  • Un collega ha fatto checkout del tuo branch per testare qualcosa e ci ha pushato sopra una correzione
  • Una pipeline CI/CD committa automaticamente sul branch (es. lint-fix, formattazione)

In sintesi: succede quando il branch non è fisicamente sotto il tuo controllo esclusivo in quel momento, anche se nominalmente è "tuo".

Da qui due strade:

SituazioneCosa fare
Push esterno recuperabile (altra tua macchina, piccola correzione)Fai git fetch origin, controlla cosa c'è di nuovo con git log origin/feature/rebase-test --oneline, eventualmente recupera i commit nuovi con cherry-pick, poi ripeti push --force-with-lease
Branch realmente condiviso, altri ci lavorano sopra attivamenteMeglio non forzare — coordina prima con chi ha pushato. Riscrivere history che altri hanno già pullato genera conflitti quando loro ripullano

Esempio completo — push esterno recuperabile

Riprendiamo lo scenario di prima: hai squashato i 3 commit in locale (f9413bd), ma nel frattempo dall'altra tua macchina avevi già pushato una piccola correzione su feature/rebase-test che ora sul remoto non hai.

git push --force-with-lease origin feature/rebase-test
! [rejected]        feature/rebase-test -> feature/rebase-test (stale info)
error: failed to push some refs to '...'

Primo passo, scarica lo stato aggiornato del remoto senza toccare il locale:

git fetch origin

Guarda cosa c'è di nuovo che il tuo locale non ha:

git log origin/feature/rebase-test --oneline
a1b2c3d (origin/feature/rebase-test) fix: corretto typo in rebase3.txt
1721453 feat: aggiunta file rebase 2
dd361f6 feat: aggiunta file rebase 1

Il commit a1b2c3d è la correzione fatta dall'altra macchina — non l'hai mai scaricata. Lo recuperi con cherry-pick sopra la tua history già squashata:

git cherry-pick a1b2c3d

Se non ci sono conflitti, ora il tuo locale ha sia lo squash che la correzione. Verifica con il log:

git log --oneline
e4f5a6b (HEAD -> feature/rebase-test) fix: corretto typo in rebase3.txt
f9413bd feat: aggiunta file rebase completo
b3aa22e (dev) feat: commit su dev nel frattempo

A questo punto puoi ripushare con sicurezza — il tuo locale contiene tutto, sia la tua history riscritta che la correzione che mancava:

git push --force-with-lease origin feature/rebase-test
Perché cherry-pick e non semplicemente un nuovo pull/merge

Un merge qui ricreerebbe la biforcazione che il rebase aveva eliminato. Il cherry-pick prende solo il contenuto del commit che ti serve e lo applica in cima alla tua history lineare, mantenendo tutto pulito e pronto per un force push coerente.

Regola pratica

Rebase interattivo e force push vanno bene su feature branch personali, non condivisi (o condivisi solo con te). Se altri ci stanno lavorando sopra, meglio evitare lo squash o farlo solo dopo essersi accordati con loro.

Prima di passare alla Fase 5 — verifica lo stato

Dopo il rebase e il push forzato, assicurati che dev locale sia aggiornato con eventuali commit avanzati durante l'esercizio:

git switch dev
git push origin dev

4.4 — I 3 scenari di --force-with-lease

Hai già visto il comportamento base di --force-with-lease. Ecco i tre scenari concreti in cui lo incontri, con comportamento e risposta diversa.

Scenario 1 — Nessuno ha pushato: il caso normale

Hai fatto rebase/squash in locale e il remoto è fermo esattamente dove lo avevi lasciato tu:

git push --force-with-lease origin feature/mio-branch
Branch 'feature/mio-branch' set up to track remote branch 'feature/mio-branch' from 'origin'.
To https://...
 + a1b2c3d...f9413bd feature/mio-branch -> feature/mio-branch (forced update)

Il push va a buon fine — Git verifica che origin/feature/mio-branch sia esattamente dove te lo aspetti (nessun commit aggiunto da altri) e sovrascrive.

Scenario 2 — Qualcuno ha pushato: push rifiutato

Nel frattempo qualcun altro (o tu da un'altra macchina) ha aggiunto commit al branch remoto:

git push --force-with-lease origin feature/mio-branch
! [rejected]        feature/mio-branch -> feature/mio-branch (stale info)
error: failed to push some refs to '...'

Il push viene bloccato. Procedi come visto nella sezione 4.3: fetch → analisi → cherry-pick dei commit mancanti → push di nuovo.

Scenario 3 — Hai fatto fetch ma non pull: attenzione

Questo è il caso più subdolo. Hai fatto git fetch origin (quindi Git sa che ci sono nuovi commit sul remoto) ma non li hai integrati. In questo caso --force-with-lease non ti protegge — sa che ci sono commit nuovi ma li sovrascrive comunque perché tecnicamente il tuo riferimento locale a origin/feature/mio-branch è aggiornato:

git fetch origin
# NON fai git pull né git rebase
git push --force-with-lease origin feature/mio-branch  # ⚠️ passa lo stesso!
--force-with-lease non è infallibile

Protegge solo se non hai ancora scaricato i nuovi commit con fetch. Se hai fatto fetch e poi forzi il push, i commit del remoto vengono sovrascritti senza avvisi. Per questo motivo, dopo un fetch, controlla sempre git log origin/feature/mio-branch --oneline prima di forzare.

SituazioneComportamento --force-with-lease
Nessuno ha pushato✅ Push riuscito — sovrascrive il remoto senza problemi
Qualcuno ha pushato, non hai fatto fetch🛑 Push rifiutato — ti protegge dal sovrascrivere il lavoro altrui
Qualcuno ha pushato, hai già fatto fetch⚠️ Push riuscito lo stesso — il lease è soddisfatto, ma perdi i commit remoti
Fase 5 — Modulo 7 del tutorial

Reset, Revert, Restore

Il kit di pronto soccorso. Come tornare indietro in sicurezza a seconda della situazione.

5.1 — git revert

Il modo sicuro per annullare un commit già pushato. Non cancella nulla: crea un nuovo commit che fa l'opposto.

git switch dev
# modifica un file, poi:
git add .
git commit -m "feat: commit sbagliato"
git push origin dev
git revert HEAD
# si apre l'editor con messaggio precompilato — lascialo e chiudi
git push origin dev
git log --oneline

Log reale ottenuto

* cb76bfb (HEAD -> dev, origin/dev) Revert "feat: commit sbagliato" * 1a8c5a7 feat: commit sbagliato * b3aa22e feat: commit su dev nel frattempo * 9697c2c Merged PR 2: feat: aggiunta navbar
Riga per riga:
* cb76bfb (HEAD -> dev, origin/dev) — il commit di revert, già pushato. origin/dev è allineato con dev = push riuscito senza problemi.

* 1a8c5a7 — il commit sbagliato originale. È ancora lì — il revert non cancella nulla dalla storia. Chiunque guardi il log vede esattamente cosa è successo: un commit sbagliato e il suo annullamento.

* b3aa22e — il commit precedente. Invariato.

Confronta con reset --hard: lì il commit sparisce dalla storia (ma rimane nel reflog). Con revert rimane tutto visibile. Questo è il comportamento corretto su branch condivisi: mai riscrivere la storia che altri potrebbero già avere in locale.
Caso delicato

Se un collega ha pushato codice che dipende dal tuo commit sbagliato, il revert rimuove le dipendenze che si aspettava e il progetto potrebbe non compilare. Parla sempre con il team prima di fare revert su branch condivisi.

5.2 — git reset (soft / mixed / hard)

Crea 3 commit di test, poi prova le 3 varianti di reset.

# modifica file, poi:
git add . && git commit -m "feat: commit A"
git add . && git commit -m "feat: commit B"
git add . && git commit -m "feat: commit C"

--soft: annulla commit, mantiene staging

git reset --soft HEAD~3
git status   # Changes to be committed
git log --oneline   # A, B, C spariti

Le modifiche sono ancora in staging area — pronte per un nuovo commit senza fare git add.

Prima di provare --mixed, ricrea i 3 commit

Dopo lo --soft i commit A, B, C non esistono più — sei tornato a un'unica modifica in staging. Per provare --mixed partendo dalla stessa situazione, prima ricommitta:

git commit -m "feat: commit A"
# modifica file, poi:
git add . && git commit -m "feat: commit B"
# modifica file, poi:
git add . && git commit -m "feat: commit C"

--mixed: annulla commit, mantiene working dir

git reset --mixed HEAD~3
git status   # Changes not staged for commit
git log --oneline   # A, B, C spariti

Le modifiche sono ancora sul disco ma fuori dalla staging area — devi fare git add prima di committare.

Prima di provare --hard, ricrea di nuovo i 3 commit

Stesso discorso: dopo --mixed i commit A, B, C sono già annullati. Ripeti la sequenza per riportarti al punto di partenza prima di provare --hard:

git add . && git commit -m "feat: commit A"
# modifica file, poi:
git add . && git commit -m "feat: commit B"
# modifica file, poi:
git add . && git commit -m "feat: commit C"

--hard: annulla commit e cancella tutto

git reset --hard HEAD~3
git status   # nothing to commit, working tree clean
git log --oneline   # A, B, C spariti

Le modifiche sono sparite anche dal disco. Nessun undo — a meno di usare il reflog.

FlagCommitStaging areaWorking directory
--softAnnullatiModifiche dentroModifiche dentro
--mixedAnnullatiModifiche fuoriModifiche dentro
--hardAnnullatiModifiche fuoriSPARITE

5.3 — git reflog

La scatola nera di Git — registra ogni operazione, anche i reset --hard. Nulla viene davvero perso per 90 giorni.

git reflog

Output reale (estratto)

cb76bfb (HEAD -> dev, origin/dev) HEAD@{0}: reset: moving to HEAD~3 96079ec HEAD@{1}: commit: feat: commit C 161e208 HEAD@{2}: commit: feat: commit B 52c4a67 HEAD@{3}: commit: feat: commit ABC cb76bfb HEAD@{4}: reset: moving to HEAD~3 0b4463f HEAD@{5}: commit: feat commit C e5a3c78 HEAD@{6}: commit: feat commit B 39cb194 HEAD@{7}: commit: feat: commit ABC cb76bfb HEAD@{8}: reset: moving to HEAD~3 ... f9413bd (origin/feature/rebase-test, feature/rebase-test) HEAD@{15}: rebase (finish): returning to refs/heads/feature/rebase-test b3aa22e HEAD@{21}: checkout: moving from feature/rebase-test to dev bcf2c4e HEAD@{26}: commit: feat: aggiunta file rebase 2 8fbaef0 HEAD@{30}: commit: feat: aggiunta file rebase 1
Riga per riga:
cb76bfb HEAD@{0}: reset: moving to HEAD~3 — l'operazione più recente: il reset --hard appena eseguito. HEAD@{0} = posizione attuale di HEAD.

96079ec HEAD@{1}: commit: feat: commit C — il commit C che sembrava perso. HEAD@{1} = dove era HEAD un'operazione fa. L'hash è ancora qui!

161e208 HEAD@{2} e 52c4a67 HEAD@{3} — i commit B e A, anch'essi recuperabili.

HEAD@{4}: reset: moving to HEAD~3 — un reset precedente (durante l'esercizio su --mixed). Il reflog registra tutto.

HEAD@{15}: rebase (finish) — il rebase della Fase 4. Il reflog mostra anche le operazioni interne del rebase.

Per recuperare il commit C: git reset --hard 96079ec. Per recuperare solo uno dei commit senza spostare HEAD: git cherry-pick 161e208.

Come recuperare un commit "perduto" — esempio completo

Hai fatto git reset --hard HEAD~3 per sbaglio e ti accorgi che il commit C conteneva del lavoro che ti serviva. Procedura:

git reflog

Trovi nel reflog l'hash del commit che vuoi recuperare (qui 96079ec, il commit C). Poi hai due strade, a seconda di cosa vuoi:

ObiettivoComando
Voglio tornare esattamente a quel punto, perdendo eventuali commit fatti dopogit reset --hard 96079ec
Voglio solo recuperare quel commit specifico senza toccare la HEAD attualegit cherry-pick 96079ec
Il reflog è solo locale

Il reflog non viene mai pushato su Azure DevOps — vive solo nella tua copia locale del repo, e per default Git lo conserva per 90 giorni. Se hai bisogno di recuperare qualcosa, fallo prima che il garbage collector di Git lo elimini definitivamente.

reset --hard vs cherry-pick: quale scegliere?

Usa reset --hard <hash> quando il commit perduto è l'ultimo di una sequenza che vuoi recuperare per intero (torni indietro nel tempo). Usa cherry-pick <hash> quando vuoi solo quel commit specifico, magari in mezzo ad altri che nel frattempo hai già rifatto correttamente — è più chirurgico e non tocca il resto della tua history attuale.

Per tornare pulito a origin/dev:

git reset --hard origin/dev

5.4 — git restore

Agisce sul singolo file, non sull'intera storia.

Scartare modifiche non ancora in staging

# modifica dashboard.txt ma NON fare git add
git restore dashboard.txt
git status   # nothing to commit

Togliere un file dalla staging area

# modifica dashboard.txt e fai git add
git restore --staged dashboard.txt
git status   # Changes not staged for commit
ComandoCosa fa
git restore <file>Scarta le modifiche dal disco
git restore --staged <file>Toglie dalla staging area, mantiene le modifiche sul disco

5.5 — git restore --source

Mentre git restore base scarta le modifiche ripristinando il file all'ultimo commit, --source permette di recuperare la versione di un file da un commit specifico, senza toccare il resto della history.

# Recupera dashboard.txt com'era 3 commit fa
git restore --source HEAD~3 dashboard.txt

# Recupera dashboard.txt da un commit specifico tramite hash
git restore --source a1b2c3d dashboard.txt

# Recupera dashboard.txt da un altro branch
git restore --source origin/dev dashboard.txt
ParteSignificato
restoreRipristina il contenuto di un file nel working directory
--sourceSpecifica da quale commit (o branch) prendere la versione del file
HEAD~33 commit indietro rispetto a HEAD — puoi usare qualsiasi hash o riferimento valido
dashboard.txtIl file da ripristinare — solo questo file viene toccato, tutto il resto rimane invariato
Il file torna in working directory, non committato

Dopo git restore --source il file viene modificato sul disco ma non messo in staging. Se vuoi committarlo devi fare git add + git commit esplicitamente. Questo ti dà la possibilità di controllare cosa è cambiato prima di committare.

Differenza con git checkout -- file (deprecato)

Il vecchio equivalente era git checkout <hash> -- nomefile.txt. git restore --source è il comando moderno che lo sostituisce — più esplicito e meno ambiguo.

Caso pratico: ho sovrascritto un file per sbaglio

# Hai sovrascritto dashboard.txt con contenuto sbagliato e già committato
# Prima trova il commit dove era corretto
git log --oneline -- dashboard.txt

# Poi recupera quella versione
git restore --source <hash-commit-corretto> dashboard.txt
git add dashboard.txt
git commit -m "fix: ripristinato dashboard.txt alla versione corretta"

Caso pratico: voglio recuperare un pezzo di codice da una versione precedente

Il caso più comune in assoluto — non vuoi fare rollback di niente, vuoi solo sbirciare una versione precedente del file, copiare quello che ti serve, e richiudere.

# Step 1 — porta sul disco la versione di 3 commit fa
git restore --source HEAD~3 dashboard.txt

# Step 2 — apri il file in IntelliJ, copia il pezzo di codice che ti serve
# (non salvare nulla, lavora solo con copia/incolla)

# Step 3 — ripristina la versione più recente del file
git restore dashboard.txt

# Step 4 — incolla il pezzo recuperato nella versione attuale e committa
git add dashboard.txt
git commit -m "fix: recuperato metodo X dalla versione di HEAD~3"
Lo Step 3 è fondamentale

git restore dashboard.txt senza --source riporta il file all'ultimo commit — cioè la versione più recente. Se salti questo step e modifichi direttamente il file "vecchio" rischi di perdere tutto il lavoro fatto nei 3 commit intermedi. Prima copi, poi ripristini, poi incolli.

In alternativa: apri il file vecchio in una seconda finestra

In IntelliJ puoi anche usare File → Open → New Window per aprire il progetto in una seconda istanza dopo aver fatto il restore, così hai le due versioni affiancate senza rischio di sovrascrivere nulla per sbaglio.

5.6 — Disaster recovery completo col reflog

Scenario peggiore: hai fatto git reset --hard su dev perdendo più commit, oppure hai fatto un rebase andato male. Il reflog è la tua ultima ancora di salvezza.

Procedura completa

# Step 1: apri il reflog
git reflog

Leggi l'output e identifica il momento prima del disastro. Ogni riga ha la forma:

<hash> HEAD@{N}: <operazione>: <descrizione>

Cerca l'ultima riga che precede l'operazione che ha causato il problema (es. reset: moving to HEAD~5 o rebase (start)).

# Step 2: verifica che l'hash trovato sia quello giusto
git show <hash>
# oppure per una vista più leggibile
git log --oneline <hash>
# Step 3a: se vuoi tornare esattamente a quel punto (perdi tutto quello che hai fatto dopo)
git reset --hard <hash>

# Step 3b: se vuoi creare un nuovo branch da quel punto (più sicuro — non tocchi dev)
git switch -c recovery/dev-backup <hash>
Preferisci sempre Step 3b in caso di dubbio

Creare un branch di recovery ti permette di confrontare la situazione prima e dopo il disastro senza rischiare di peggiorare le cose. Una volta verificato che il branch di recovery contiene tutto quello che ti serve, puoi fare il reset su dev con più tranquillità.

# Step 4: se hai recuperato su un branch di recovery, riporta dev lì
git switch dev
git reset --hard recovery/dev-backup

# Step 5: pusha con force-with-lease per allineare il remoto
git push --force-with-lease origin dev
force-with-lease su dev è una cosa seria

Se altri hanno già pullato da dev nel frattempo, dovranno fare un git pull --force o un reset per allinearsi alla nuova history. Coordina sempre con il team prima di forzare su branch condivisi come dev.

Cherry-pick 🍒

Prendi un commit specifico da qualsiasi punto della storia e applicalo dove vuoi.

6.1 — Cherry-pick base

Crea 3 commit distinti su file diversi, poi porta su dev solo il commit B.

git switch dev
git switch -c feature/cherry-test
# crea commit-a.txt, poi:
git add . && git commit -m "feat: commit A - file alpha"
# crea commit-b.txt, poi:
git add . && git commit -m "feat: commit B - file beta"
# crea commit-c.txt, poi:
git add . && git commit -m "feat: commit C - file gamma"
git log --oneline   # prendi nota degli hash

Log del feature branch

* 58cea56 (HEAD -> feature/cherry-test) feat: commit A - file gamma * 08a286e feat: commit A - file beta * cb59bf6 feat: commit A - file alpha * df53ee5 (origin/dev, dev) spostato dashboard di cartella
git switch dev
git cherry-pick 08a286e   # l'hash del commit B
git log --oneline

Log di dev dopo il cherry-pick

* 6488e18 (HEAD -> dev) feat: commit A - file beta * df53ee5 (origin/dev) spostato dashboard di cartella * 9697c2c Merged PR 2: feat: aggiunta navbar
Riga per riga (log di dev dopo il cherry-pick):
* 6488e18 (HEAD -> dev) — il commit copiato da cherry-pick. L'hash è diverso da 08a286e del feature branch — è un commit completamente nuovo, con lo stesso contenuto ma con un genitore diverso (df53ee5 invece di cb59bf6).

* df53ee5 (origin/dev) — il commit precedente di dev. origin/dev è rimasto qui = non hai ancora pushato.

*   9697c2c — la PR 2. Più vecchio, invariato.

Cosa NON vedi: i commit A (cb59bf6) e C (58cea56) di feature/cherry-test. Non sono stati toccati — il cherry-pick ha preso solo il commit B e lo ha replicato su dev.
Pusha prima di proseguire

Il cherry-pick ha creato un nuovo commit solo in locale — allinea il remoto prima di passare all'esempio successivo:

git push origin dev

6.2 — Cherry-pick con conflitto

git switch dev
# modifica dashboard.txt, poi:
git add . && git commit -m "feat: modifica dashboard su dev"
git switch feature/cherry-test
# modifica la STESSA riga di dashboard.txt, poi:
git add . && git commit -m "feat: modifica dashboard su feature"
git log --oneline   # prendi nota dell'hash di questo commit

Output del cherry-pick in conflitto

Risolvi il file a mano (rimuovi i marcatori), poi:

git add .

Cherry-pick come strumento di recupero

Se hai fatto un reset --hard e vuoi recuperare solo uno dei commit "persi" (visibili nel reflog), puoi usare git cherry-pick <hash> invece di spostare tutta la HEAD con reset --hard.
