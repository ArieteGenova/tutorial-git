Modulo 1 · Basi

Le fondamenta di Git

Cosa succede davvero dentro Git quando esegui un comando — oggetti interni, hash SHA-1, il DAG e le tre aree di lavoro.

Cos'è Git, davvero

Git non è un sistema di backup e non è un sistema di sincronizzazione file. È un database di snapshot immutabili del tuo progetto, navigabile in ogni direzione nel tempo. Ogni volta che fai un commit, Git fotografa l'intero stato del progetto e salva quella foto in modo permanente.

Git è distribuito: non esiste un server centrale indispensabile — ogni clone è un repository completo con tutta la storia. Il remote (es. Azure DevOps) è semplicemente un'altra copia, che per convenzione usiamo come punto di sincronizzazione.

Cosa significa "snapshot"

Uno snapshot è una fotografia dello stato completo dei file del progetto in un preciso istante nel tempo. Quando fai un commit, non stai dicendo "salva le differenze" come farebbe un sistema di patch. Stai dicendo: "questo è l'intero stato del mio progetto in questo momento."

Ma allora Git duplica tutti i file ad ogni commit?

No — ed è qui che Git è intelligente. Dal punto di vista concettuale ogni commit è uno snapshot completo, ma internamente Git riutilizza gli oggetti che non sono cambiati. Nell'esempio sopra, Program.cs e appsettings.json non sono cambiati: il commit B punta semplicemente agli stessi oggetti del commit A. Solo Cliente.cs genera un nuovo oggetto.

Il commit B non dice "aggiungi la riga string Cognome;". Dice: "Cliente.cs adesso è questo contenuto completo". Il modello mentale corretto di Git non è una serie di modifiche ai file, ma una serie di istantanee complete del progetto nel tempo.

Modello concettuale vs ottimizzazione su disco

All'inizio Git salva i file come blob completi (loose objects). Successivamente, durante operazioni come git gc, può creare dei packfile in cui comprime gli oggetti e memorizza alcune versioni come delta rispetto ad altre — esattamente come un album fotografico che in archivio conserva una foto intera e per le altre tiene solo le differenze, per risparmiare spazio. Per te sviluppatore però ogni commit continua a comportarsi come una fotografia completa e immutabile. Questa distinzione tra modello logico (snapshot) e ottimizzazione fisica (packfile con delta) è una delle idee più importanti per capire Git in profondità.

Le tre aree di lavoro

Tutto il lavoro in Git ruota attorno a tre zone ben distinte:

[ Working tree ] → git add → [ Staging area / index ] → git commit → [ Repository .git/ ]

I tuoi file su disco                  Foto preparatoria                           Database oggetti

Working tree — i tuoi file su disco, quelli che vedi e modifichi nell'IDE. È il tuo tavolo da lavoro.

Staging area (index) — la zona di preparazione dove metti i file che vuoi includere nel prossimo commit tramite git add. Ti permette di costruire commit chirurgici.

Repository locale (.git/) — il database degli oggetti dove i commit vengono cristallizzati in modo permanente. Tutto questo è esclusivamente locale — il remote entra in gioco solo con git push.

Ricorda

Fino al git push sei sempre e solo in locale. Puoi fare mille commit offline — nessun server coinvolto.

I quattro oggetti interni

Git è fondamentalmente un key-value store. La chiave è l'hash SHA-1, il valore è il contenuto dell'oggetto. Esistono quattro tipi:

TipoCosa rappresenta
blobIl contenuto grezzo di un singolo file — zero metadati, zero nome file, solo bytes
treeUna directory: lista di blob e altri tree con permessi e nomi
commitUno snapshot completo + metadati (autore, timestamp, messaggio) + puntatore al commit padre
tagUn'etichetta annotata su un commit, con metadati propri
Cosa importante sui blob

Se due file hanno lo stesso contenuto, Git salva un solo blob. Il nome del file non esiste nel blob — esiste solo nel tree che lo referenzia.

Gli hash SHA-1

Ogni oggetto viene identificato da un hash SHA-1 — una stringa esadecimale di 40 caratteri tipo a3f5c2d1e4b6.... Questo hash è il fingerprint del contenuto: se il contenuto cambia di un singolo byte, l'hash cambia completamente. Questo garantisce l'integrità assoluta della storia.

Git abbrevia l'hash ai primi 7 caratteri nella maggior parte dei comandi — abbastanza unico da non creare ambiguità in progetti normali.

Il DAG — la struttura che governa tutto

La storia di Git è un DAG (Directed Acyclic Graph — grafo orientato aciclico). Ogni commit punta al suo commit padre. Non si va mai "all'indietro" — il grafo fluisce sempre in una direzione.

[a3f5c2d] ←── [7b2e91f] ←── [9d1a4bc] ←── [f4c8e3a]
Init            Login           Fix bug          Feature       ← [main] ← [HEAD]

Un branch è solo un file con 40 caratteri dentro — l'hash dell'ultimo commit.
Insight fondamentale

Un branch non è una copia del codice. È semplicemente un file di testo in .git/refs/heads/ che contiene l'hash dell'ultimo commit. Per questo creare un branch in Git è istantaneo.

Cosa sono i branch (davvero)

Un branch non è una copia del codice. È un file di testo in .git/refs/heads/ che contiene l'hash dell'ultimo commit. Creare un branch è istantaneo e pesa quasi zero — Git sposta solo un puntatore.

Ogni volta che fai un commit su un branch, quel puntatore avanza automaticamente all'ultimo commit. HEAD è il puntatore speciale che ti dice dove sei tu adesso.

[a3f5c2d] ←── [7b2e91f] ←── [9d1a4bc]
                                                  ↑ [master] ← [HEAD]

[a3f5c2d] ←── [7b2e91f] ←── [9d1a4bc] ←── [c7f1d2e]
                                                                                ↑ [feature/login] ← [HEAD]

Dopo git switch -c feature/login e un commit: master e feature/login puntano a commit diversi.

Fast-forward vs Merge commit

La domanda chiave è una sola: il branch di destinazione si è mosso mentre lavoravi sul tuo branch? La risposta determina tutto.

CASO 1 — Fast-forward (nessun collega ha toccato dev/master)

Crei feature/login da dev e fai due commit. Nel frattempo nessuno tocca dev — rimane fermo al commit di partenza.

Fai git pull — non cambia niente, dev è già aggiornato.
Fai git merge feature/login — Git sposta semplicemente il puntatore di dev in avanti. Nessun merge commit, storia lineare:

Se vuoi che nella storia rimanga traccia del branch, forzi il merge commit con --no-ff:

git switch dev
git pull
git merge --no-ff feature/login
git push

CASO 2 — Merge commit automatico (un collega ha pushato su dev)

Crei feature/login e fai due commit. Nel frattempo un collega pusha C4 su dev — origin è avanti rispetto al tuo locale.

Fai git pull — il tuo dev locale si aggiorna con il commit del collega. Ora le storie sono divergenti:

Git vede due storie divergenti — il fast-forward è impossibile. Crea automaticamente un merge commit M senza bisogno di --no-ff:

git switch dev
git pull                   # allinei dev locale con origin
git merge feature/login    # --no-ff è ridondante qui, Git crea M da solo
git push                   # porti il risultato su origin
Regola pratica

La domanda da farti prima del merge è sempre: dev/master si è mosso mentre lavoravo?
— No → fast-forward possibile → se vuoi lasciare traccia del branch usa --no-ff
— Sì → storie divergenti → merge commit automatico, --no-ff è ridondante

In ogni caso fai sempre git pull prima del merge — non saltarlo mai.

Gestione dei conflitti

Un conflitto nasce quando due branch hanno modificato le stesse righe dello stesso file in modo diverso. Git mette in pausa il merge e marca i file così:

<<<<<<< HEAD
versione del branch corrente (master)
=======
versione del branch che stai mergiando
>>>>>>> feature/login

Il processo per risolvere è sempre questo:

1. Apri il file in conflitto — IntelliJ e VS Code mostrano i marker visivamente con pulsanti per scegliere.

2. Decidi quale versione tenere, o scrivi una versione ibrida che incorpora entrambe.

3. Rimuovi tutti i marker (<<<<<<<, =======, >>>>>>>). Se ne lasci uno, il commit fallisce.

4. Fai git add sul file risolto.

5. Concludi con git commit — Git precompila il messaggio automaticamente.

Attenzione

Se la risoluzione si complica e vuoi ricominciare da capo: git merge --abort riporta tutto allo stato precedente al merge, senza danni. Usalo senza esitazione.

Pull Request su Azure DevOps

In un team reale non si fa il merge direttamente da riga di comando — si usa una Pull Request (PR). È il meccanismo per proporre l'integrazione di un branch, fare code review e discutere le modifiche prima che entrino in master.

1. Push del branch — git push -u origin feature/nome-branch — rende il branch visibile su Azure DevOps.

2. Apertura PR — Repos → Pull Requests → New Pull Request. Scegli il branch sorgente e quello di destinazione (master o main).

3. Code review — i colleghi commentano le singole righe. Puoi pushare nuovi commit sullo stesso branch, la PR si aggiorna automaticamente.

4. Approvazione — quando i reviewer approvano, il pulsante Complete PR diventa disponibile.

5. Merge — Azure DevOps fa il merge (di default con merge commit) e può cancellare il branch sorgente automaticamente. (→ Modulo 5: i tre tipi di merge)

Branch Policies

Su Azure DevOps puoi impostare regole sul branch master: richiedere almeno un reviewer, bloccare i push diretti senza PR, richiedere che la pipeline CI sia verde. Si configurano in Repos → Branches → tasto destro sul branch → Branch policies.

Cosa succede nel repository locale dopo una PR

Una volta completata la PR su Azure DevOps, torna nel terminale e allinea il tuo locale:

Modulo 3 · Basi

Branch e HEAD

Cosa sono i branch · HEAD · checkout / switch · branch locali e remoti

Cosa sono i branch — sotto il cofano

Nel Modulo 2 abbiamo usato i branch. Adesso capiamo cosa sono davvero, perché questo cambia il modo in cui li usi e li pensi.

Un branch non è una cartella, non è una copia del progetto. È semplicemente un file di testo dentro .git/refs/heads/ che contiene 40 caratteri: l'hash SHA-1 dell'ultimo commit su quel branch. Niente di più.

# Puoi vederlo direttamente:
cat .git/refs/heads/master
9d1a4bc3f8e2a1d0c7b5e4f6a2d8c9b1e3f5a7d9

# Ed è identico a quello che ti mostra git log:
git log --oneline -1
9d1a4bc feat: primo commit

Ecco perché creare un branch in Git è istantaneo e gratuito — Git scrive 40 caratteri in un file. Nessuna copia, nessuna duplicazione.

Perché questo conta

In altri sistemi di versioning (SVN, per esempio) un branch era una vera copia della directory. Costoso, lento, pesante. In Git è un puntatore — puoi crearne quanti vuoi senza nessun overhead. Questa è una delle ragioni per cui il branching frequente è una buona pratica in Git, non un problema.

HEAD — il puntatore che ti dice dove sei

HEAD è il puntatore speciale di Git. Ti indica sempre su quale commit stai lavorando adesso. Nella situazione normale, HEAD non punta direttamente a un commit, ma a un branch — e quel branch punta al commit. Si chiama HEAD collegato (attached HEAD).

Quando fai un nuovo commit, Git avanza automaticamente il puntatore del branch corrente — e HEAD lo segue. Non devi fare nulla tu.

Puoi sempre controllare dove punta HEAD con:

cat .git/HEAD
ref: refs/heads/master   # HEAD collegato a master

HEAD staccato (detached HEAD)

Esiste una situazione particolare che può sorprenderti: il detached HEAD. Succede quando HEAD punta direttamente a un commit specifico invece che a un branch. Tipicamente lo ottieni facendo:

# Spostarsi su un commit specifico (non su un branch)
git switch --detach 9d1a4bc

# Oppure con la sintassi storica:
git checkout 9d1a4bc

In questa modalità puoi guardare il codice ma fai attenzione se committi: i nuovi commit non appartengono a nessun branch. Se poi fai git switch master, quei commit diventano irraggiungibili e Git li rimuoverà col tempo.

Detached HEAD — cosa fare

Se ti trovi in detached HEAD per esplorare il codice: nessun problema, guarda pure. Se hai committato qualcosa che vuoi salvare, crea subito un branch: git switch -c nome-branch. Questo aggancia i commit a un branch reale e li salva.

Branch locali e branch remoti

Git distingue due tipi di branch. Capire la differenza evita molta confusione.

Branch locali

Esistono solo sul tuo computer, in .git/refs/heads/. Li crei con git switch -c, ci lavori, ci committi. Finché non fai git push, il remote non sa che esistono.

Branch remoti (tracking branch)

Sono copie locali di sola lettura dello stato del remote. Vivono in .git/refs/remotes/origin/ e si aggiornano solo quando fai git fetch o git pull. Non ci lavori direttamente — sono un riferimento.

git branch        # solo branch locali
git branch -a     # locali + remoti (i remoti hanno prefisso remotes/origin/)
git branch -r     # solo i branch remoti
Quello che vedi con git branch -a: * feature/login ← branch locale, quello su cui sei master ← branch locale remotes/origin/master ← copia remota di master remotes/origin/dev ← copia remota di dev

Il tracking — la relazione locale ↔ remoto

Quando un branch locale "traccia" un branch remoto, Git sa dove fare push e pull senza che tu debba specificarlo ogni volta. Si imposta con il flag -u al primo push:

git push -u origin feature/login
# Dopo questo, per tutti i push successivi basta:
git push

Divergenza locale/remota

Una situazione frequente in team: hai committato in locale e nel frattempo un collega ha pushato sul remote. I due branch sono divergenti — hanno storie diverse che partono dallo stesso punto.

In questo caso un semplice git push viene rifiutato. Le opzioni sono:

# Opzione 1 — fetch + merge (più sicuro, lascia traccia della divergenza)

git merge origin/master

# Opzione 2 — pull (fa fetch + merge in un colpo solo)
git pull

# Opzione 3 — fetch + rebase (storia più lineare, → Modulo 6: Rebase)

L'opzione 3 con git rebase produce una storia più pulita rispetto al merge, ma richiede di capire come funziona il rebase per usarlo correttamente. (→ Modulo 6: il rebase in dettaglio)

Se ti trovi a volerlo fare, fermati e valuta se non sia meglio un merge o un rebase. Il --force-with-lease è un'alternativa meno distruttiva — ne parleremo nei moduli avanzati.

Remote, push e pull

Cos'è un remote

Un remote è semplicemente un alias che punta a un altro repository Git — nella maggior parte dei casi su Azure DevOps, GitHub o simili. Non è magia: è solo un URL memorizzato nella config del tuo repo.

Quando fai git clone, Git fa due cose insieme: copia tutti gli oggetti del repository remoto in locale (commit, blob, tree — l'intero database), e registra l'URL di provenienza sotto il nome origin. Quel nome è letteralmente salvato in .git/config:

[remote "origin"]
    url = https://dev.azure.com/org/progetto/_git/repo
    fetch = +refs/heads/*:refs/remotes/origin/*

Da quel momento, ogni volta che Git vede origin/main o origin/feature/login, sa che sta parlando dello stato di quei branch sull'URL registrato sotto quel nome. È un alias — niente di più. Questo è anche il motivo per cui i tracking branch hanno quel prefisso: la parte prima dello slash è il nome del remote, la parte dopo è il branch.

origin non è speciale

origin è solo un nome convenzionale assegnato automaticamente da git clone. Potresti chiamarlo azure o server — funzionerebbe uguale. La convenzione è così consolidata che vale sempre la pena rispettarla.

Puoi avere più remote sullo stesso repo locale — ma nella pratica quotidiana lavorerai quasi sempre solo con origin.

git remote -v                          # mostra tutti i remote configurati
git remote add upstream <url>         # aggiunge un secondo remote
git remote remove upstream            # rimuove un remote
origin non è speciale

origin è solo un nome convenzionale. Potresti chiamarlo azure o server — funzionerebbe uguale. La convenzione è così consolidata che vale sempre la pena rispettarla.

La differenza tra fetch, pull e push

Questi tre comandi gestiscono la sincronizzazione con il remote. Capire cosa toccano e cosa non toccano ti salva da brutte sorprese.

git fetch

Scarica i nuovi oggetti dal remote ma non tocca nulla di locale. I tuoi file, il tuo working tree, il tuo branch corrente: tutto rimane esattamente com'era. Git aggiorna solo i puntatori remoti (origin/main, origin/feature/x, ecc.).

git fetch                  # scarica da origin (default)
git fetch origin           # esplicito — identico al precedente
git fetch --all            # scarica da tutti i remote configurati

Dopo un git fetch puoi ispezionare cosa c'è sul remote prima di integrare:

git log origin/main --oneline    # vedi i commit arrivati dal remote
git diff HEAD origin/main        # differenza tra il tuo locale e il remote
Fetch prima di merge: buona abitudine

Prima di fare merge o rebase, un git fetch ti dà una visione aggiornata di quello che c'è sul remote — senza rischiare nulla. È il modo più sicuro per capire se il tuo branch è ahead, behind o divergente.

git pull

git pull è un'operazione in due fasi: prima fa un fetch, poi integra i nuovi commit nel branch corrente. Di default l'integrazione avviene tramite merge.

git pull                   # fetch + merge (default)
git pull --rebase          # fetch + rebase (storia più lineare)
git pull origin main       # pull esplicito da un branch specifico
pull --rebase: quando usarlo (→ Modulo 6 per i dettagli)

git pull --rebase riscrive i tuoi commit locali sopra quelli arrivati dal remote, evitando merge commit spurious. È ottimo per mantenere la storia pulita — ma non usarlo su commit già pushati su un branch condiviso. Ne parleremo in dettaglio nel Modulo 6 → git pull --rebase.

git push

Invia i tuoi commit locali al remote. Se stai pushando un branch per la prima volta, devi specificare il remote e impostare il tracking con -u:

git push -u origin feature/login   # primo push — imposta il tracking
git push                           # push successivi (tracking già impostato)
git push origin feature/login      # push esplicito senza -u (tracking non si imposta)

feature/loginil branch locale che vuoi pushare — verrà creato sul remote con lo stesso nome

I tracking branch

Quando imposti il tracking con -u, Git crea un collegamento tra il tuo branch locale e il corrispondente branch remoto. Questa coppia si chiama tracking branch.

feature/login (locale) → traccia → origin/feature/login (remoto) main (locale) → traccia → origin/main (remoto)

Grazie al tracking, Git può dirti in ogni momento se sei ahead (hai commit non ancora pushati), behind (il remote ha commit che non hai ancora) o divergente (entrambe le cose).

La divergenza locale/remota

La situazione più delicata è quando la storia diverge: tu hai fatto commit in locale, ma nel frattempo qualcuno ha pushato sul remote. I due rami della storia si sono separati.

In questa situazione git push viene rifiutato con un errore tipo:

! [rejected]  main → main (fetch first)
error: failed to push some refs to 'https://...'
hint: Updates were rejected because the remote contains work that you do not have locally.

Git ti sta dicendo: "non ti lascio sovrascrivere il remote perché perderesti il commit C4 del tuo collega." La soluzione corretta è integrare prima i cambiamenti del remote:

git fetch                  # aggiorna origin/main senza toccare nulla
git merge origin/main      # integra i commit del remote nel tuo branch
git push                   # ora il push va a buon fine

# Oppure in un unico comando:
git pull                   # fetch + merge automatico
git push
Mai fare push --force su branch condivisi

git push --force forza il push sovrascrivendo la storia remota — e con essa i commit dei colleghi. Su branch condivisi (main, dev, release) è quasi sempre un disastro. Se ti trovi a volerlo fare, fermati: probabilmente la soluzione giusta è un merge o un pull. L'unica alternativa meno distruttiva è --force-with-lease, che fallisce se il remote ha commit che non conosci — ne parleremo nei moduli avanzati.

Merge e conflitti

I tre tipi di merge a colpo d'occhio

Prima di entrare nell'algoritmo, ecco una mappa visiva dei tre modi in cui Git (e Azure DevOps) può unire due branch:

I comandi esatti per ogni tipo di merge

Il Modulo 2 ha già introdotto fast-forward e no-ff — qui li ritrovi tutti e tre in un posto solo, con il comando preciso per ciascuno.

1 · Merge commit — storie divergenti, Git crea M automaticamente:

git switch dev
git merge feature/login
# Le storie hanno divergito → Git crea il merge commit M da solo
# --no-ff qui è ridondante ma non fa danni

1b · Merge commit forzato (no-ff) — storie non divergenti ma vuoi lasciare traccia visiva del branch:

git switch dev
git merge --no-ff feature/login
# Forza la creazione del merge commit M anche se il fast-forward era possibile
# Nel log si vede chiaramente che feature/login è esistito come branch separato
# Azure DevOps fa questo automaticamente quando completi una PR

2 · Fast-forward — dev non ha nuovi commit, il puntatore si sposta semplicemente:

git switch dev
git merge feature/login
# Se dev non ha commit dopo il punto di divergenza → fast-forward automatico
# Nessun merge commit, storia lineare — non si vede il confine del branch

3 · Squash merge — tutti i commit del branch compressi in uno solo:

git switch dev
git merge --squash feature/login
# --squash mette tutto in staging ma NON fa il commit automaticamente
git commit -m "feat: implementa sistema di login"
Come si relaziona --no-ff con i tre tipi

--no-ff non è un quarto tipo di merge — è un modificatore del merge commit. Serve solo quando le storie non sono divergenti (fast-forward sarebbe possibile) ma vuoi comunque un merge commit per preservare la traccia visiva del branch. Quando le storie sono già divergenti, Git crea il merge commit da solo e --no-ff è ridondante. Azure DevOps usa sempre --no-ff implicitamente quando completi una PR.

Come Git decide come fare il merge

Nel Modulo 3 hai visto fast-forward e merge commit nella pratica quotidiana. Qui andiamo più in profondità: capire l'algoritmo che Git usa internamente è quello che ti permette di gestire i conflitti con lucidità invece che con ansia.

Cos'è la merge base

Prima di tutto il concetto chiave: la merge base è semplicemente l'ultimo commit che hai in comune con l'altro branch — il punto esatto in cui le due storie si sono separate.

Quando fai git merge feature/login, Git esegue sempre la stessa sequenza logica:

1. Trova la merge base (C) — l'antenato comune più recente tra i due branch.

2. Confronta dev con C → "cosa ha cambiato dev dopo la separazione?" (il commit D).

3. Confronta feature/login con C → "cosa ha cambiato feature dopo la separazione?" (i commit E e F).

4. Combina le due serie di modifiche. Righe diverse → merge automatico. Stesse righe → conflitto.

Questo approccio si chiama 3-way merge — tre punti: dev, feature/login, e la merge base C nel mezzo.

Perché la merge base è fondamentale

Senza un punto di riferimento comune, Git non saprebbe chi ha cambiato cosa. Grazie alla merge base riesce a ragionare: "questa riga l'ha modificata solo feature, quindi la accetto senza problemi; quest'altra l'hanno modificata entrambi in modo diverso, quindi è un conflitto e tocca decidere a te."

Esempio concreto

Nel commit C (merge base), appsettings.json aveva:

"timeout": 30

Dopo la separazione:

— Su dev qualcuno ha cambiato il valore a 60

— Su feature/login tu l'hai cambiato a 120

Git guarda la merge base e vede: entrambi hanno modificato questa riga rispetto all'originale, in modo diverso → conflitto. Deve chiedere a te.

Se invece solo tu avessi modificato quella riga e dev non l'avesse toccata → Git accetta la tua modifica in silenzio, perché è chiaro chi ha fatto cosa.

Non devi calcolarla a mano

Nella pratica quotidiana Git trova la merge base da solo — non devi pensarci. Ma capire che esiste ti spiega perché certi merge vanno lisci e certi no, e ti dà il contesto giusto per leggere i conflitti senza farsi prendere dal panico.

Trovare la merge base manualmente

Puoi chiederlo direttamente a Git — utile per capire da dove parte il contesto di un conflitto:

git merge-base HEAD feature/x
# output: hash del commit antenato comune più recente, es. a3f5c2d

git merge-base --all HEAD feature/x
# se ci sono più merge base (storia complessa), le mostra tutte

Puoi anche vedere le differenze rispetto alla merge base prima di fare il merge — preventivo utilissimo in team:

# Cosa ha cambiato feature/x rispetto al punto di divergenza
git diff $(git merge-base HEAD feature/x) feature/x

# Oppure con la notazione triplo punto (più rapida)
git diff HEAD...feature/x

Workflow completo di risoluzione conflitto

Situazione: hai fatto git merge feature/login e Git si è fermato con conflitti. Ecco il processo corretto passo per passo.

Step 1 — Capire lo stato del merge

git status
# Output tipico:
On branch dev
You have unmerged paths.
  (fix conflicts and run "git commit")

Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   src/Services/AuthService.cs
        both modified:   appsettings.json

I file marcati come both modified sono quelli in conflitto. git diff ti mostra tutti i marker in un colpo solo:

git diff
# mostra tutti i blocchi in conflitto in tutti i file

Step 2 — Scegli lo strumento di risoluzione

Hai tre opzioni, dalla più semplice alla più potente:

VS Code / IntelliJ — mostrano i marker con pulsanti visivi "Accept Current / Accept Incoming / Accept Both". Per il 90% dei conflitti è la via più rapida.

git mergetool — apre un tool configurabile che mostra le tre versioni in parallelo. Git non ha un tool di default fisso: scorre una lista di tool predefiniti e usa il primo che trova installato nel sistema (opendiff, kdiff3, meld, vimdiff e altri). Su Windows senza configurazione esplicita di solito finisce su vimdiff.

# Vedi quale tool userebbe Git sul tuo sistema
git mergetool --tool-help

# Usa un tool solo per questa volta (temporaneo)
git mergetool --tool=vscode

# Imposta VS Code come default permanente per tutti i repository
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait $MERGED'

# Imposta IntelliJ IDEA come default permanente
# (IntelliJ non è built-in in Git: va configurato il comando completo)
git config --global merge.tool intellij
git config --global mergetool.intellij.cmd '"C:/Program Files/JetBrains/IntelliJ IDEA <versione>/bin/idea64.exe" merge "$LOCAL" "$REMOTE" "$BASE" "$MERGED"'
git config --global mergetool.intellij.trustExitCode true

git config merge.tool vscodeImposta VS Code come default permanente solo nel repository corrente

VS Code è un tool built-in di Git — basta specificare il nome e funziona. IntelliJ non è built-in: devi fornire tu il percorso completo dell'eseguibile con mergetool.intellij.cmd, altrimenti Git non sa come avviarlo. Se lavori principalmente da IntelliJ, il workflow più comodo è usare direttamente Git → Resolve Conflicts dall'IDE invece di passare per git mergetool.

Modifica manuale — apri il file, leggi i marker, scrivi tu la versione finale. Necessario quando la risoluzione corretta non è né l'una né l'altra versione ma una combinazione.

Step 3 — Risolvi ogni file

Per ogni file in conflitto:

1. Apri il file e leggi entrambe le versioni con attenzione — non scegliere a casaccio.

2. Scrivi la versione finale corretta (potrebbe essere A, B, o un mix).

3. Rimuovi tutti i marker — <<<<<<<, |||||||, =======, >>>>>>>. Se ne lasci uno, il commit fallisce.

4. Salva il file.

Step 4 — Segna come risolto e concludi

Una cosa che confonde spesso: il git commit finale non rifà il merge. Il merge è già a metà — Git lo ha avviato con il tuo git merge, ha applicato tutto quello che poteva da solo, e si è fermato solo sui conflitti. Dentro .git/ esiste in questo momento un file chiamato MERGE_HEAD che dice a Git "c'è un merge in sospeso, aspetto che l'utente finisca".

Quando fai git add sui file risolti stai dicendo a Git "questo conflitto è sistemato". Quando fai git commit stai semplicemente chiudendo il merge già iniziato — Git lo sa grazie a MERGE_HEAD, e per questo precompila automaticamente il messaggio con "Merge branch 'feature/login' into dev". Dopo il commit, MERGE_HEAD sparisce e il merge è completo.

# Dopo aver risolto ogni file in conflitto:
git add src/Services/AuthService.cs
git add appsettings.json

# Verifica che non ci siano altri file in conflitto
git status

# Chiude il merge già iniziato — Git precompila il messaggio automaticamente
git commit
# oppure, se vuoi personalizzare il messaggio:
git commit -m "merge: integra feature/login in dev"
Quando la risoluzione si complica: abort subito

Se vuoi annullare tutto e ripartire da zero, git merge --abort cancella MERGE_HEAD e riporta tutto allo stato esatto di prima del merge — zero danni. Poi analizza con calma, magari chiama un collega, e riparti. È molto meglio che committare una risoluzione sbagliata.

Conflitti comuni e come risolverli

Caso 1 — Stessa riga, intento diverso

Tu hai cambiato una costante per prod, il collega per staging. La risoluzione dipende dal branch di destinazione: se stai mergiando in dev vuoi la versione staging; se stai mergiando in main vuoi la versione prod. Leggi il contesto prima di scegliere.

Caso 2 — File rinominato da uno, modificato dall'altro

Git lo segnala come rename/modify conflict. In questo caso --ours e --theirs sono fuorvianti — non si riferiscono a persone, ma a ruoli fissi nel merge. (→ Modulo 6: come cambiano durante il rebase):

--ours = il branch su cui sei quando lanci git merge — cioè HEAD.
--theirs = il branch che stai mergiando — il branch sorgente.

Esempio: sei su dev e fai git merge feature/login → --ours è dev, --theirs è feature/login. Il problema è che potresti essere tu l'autore del codice su feature/login — quindi --theirs sarebbe in realtà il tuo lavoro. Il nome suggerisce "roba degli altri" ma non è detto che sia così.

Regola pratica: prima di usare --ours o --theirs chiediti sempre "su quale branch sono adesso?" — quello è --ours, l'altro è --theirs. Ignora chi ha scritto il codice, guarda solo la posizione dei branch. Il risultato concreto dipende inoltre da chi ha rinominato e chi ha modificato, quindi senza guardare prima il contenuto rischi di prendere la versione sbagliata.

Il modo più sicuro è guardare prima entrambe le versioni e poi decidere. Esempio: sei su dev e hai fatto git merge feature/login:

# Guarda come è il file su dev (HEAD)
git show HEAD:src/NomeFileOriginale.cs

# Guarda come è il file su feature/login (branch sorgente)
git show MERGE_HEAD:src/NomeFileOriginale.cs

# Vuoi tenere la versione del branch su cui sei (HEAD)?
# Sovrascrive il file con la versione di dev — la versione di feature/login viene ignorata
git checkout --ours -- src/NomeFileOriginale.cs

# Vuoi tenere la versione del branch sorgente?
# Sovrascrive il file con la versione di feature/login — la versione di dev viene ignorata
git checkout --theirs -- src/NomeFileOriginale.cs

# In entrambi i casi, segna il file come risolto
git add src/NomeFileOriginale.cs

Caso 3 — File cancellato da uno, modificato dall'altro

Git si ferma con deleted by us / deleted by them. Devi decidere: tieni il file o lo elimini definitivamente?

# Vuoi tenere il file
git add NomeFile.cs

# Vuoi eliminarlo definitivamente
git rm NomeFile.cs

Caso 4 — Conflitto su file binari (immagini, .dll, .pdf)

Git non può fare 3-way merge su file binari — ti chiede di scegliere una versione o l'altra:

# Tieni la tua versione
git checkout --ours -- assets/logo.png
git add assets/logo.png

# Tieni quella del collega
git checkout --theirs -- assets/logo.png
git add assets/logo.png

Strategie di merge avanzate

Git supporta diverse strategie di merge — nella pratica quotidiana non le cambi quasi mai, ma capirle ti aiuta a risolvere situazioni particolari.

StrategiaQuando si usa

recursiveDefault nelle versioni precedenti — ancora disponibile. Con -X ours o -X theirs risolve automaticamente i conflitti preferendo una delle due versioni.

L'opzione più utile in pratica è -X (extended option) sulla strategia di default:

# In caso di conflitto, preferisci sempre la tua versione (HEAD)
git merge -X ours feature/x

# In caso di conflitto, preferisci sempre la versione del branch sorgente
git merge -X theirs feature/x
Usa -X con criterio

-X ours e -X theirs risolvono tutti i conflitti automaticamente in un senso — senza mostrarteli. Utile in script o situazioni molto controllate, ma in un merge normale rischi di perdere modifiche legittime del collega senza accorgertene.

Squash merge — quando e perché

Azure DevOps e GitHub offrono una terza opzione oltre a merge commit e fast-forward: lo squash merge. Invece di portare tutti i commit del branch, li comprime in un singolo commit pulito sul branch di destinazione. (→ Modulo 6: Squash e cherry-pick non vanno d'accordo)

# Squash merge da riga di comando
git switch dev
git merge --squash feature/login
# --squash mette tutto in staging ma NON fa il commit automaticamente
git commit -m "feat: implementa sistema di login"

Su Azure DevOps lo trovi nel menu a tendina del pulsante "Complete" nella Pull Request come opzione Squash commit.

La scelta dipende dal team

Non esiste una risposta universale. Molti team usano squash merge su main per tenere la storia pulita e merge commit su dev per tenere la storia completa. L'importante è che il team si accordi su una convenzione e la segua.

Prevenire i conflitti — best practice in team

La risoluzione dei conflitti è importante, ma la prevenzione lo è ancora di più.

Sincronizza spesso. Più aspetti a sincronizzare il tuo branch con dev, più le storie divergono e più conflitti accumuli. Ogni mattina, prima di iniziare, assicurati di essere sul tuo branch e poi:

git switch feature/login   # assicurati di essere sul tuo branch
git fetch                  # scarica gli aggiornamenti dal remote
git merge origin/dev       # porta i commit di dev nel tuo branch

Nota: git fetch + git merge origin/dev è equivalente a git pull origin dev — la forma estesa è preferibile in un contesto didattico perché rende espliciti i due passi separati.

Tieni però presente una conseguenza: quando mergi origin/dev nel tuo branch, i commit dei colleghi entrano nella storia del tuo branch. Non è un problema per il merge finale — quei commit esistono già su dev, non stai introducendo niente di nuovo. Ma se in futuro volessi fare cherry-pick dei tuoi commit, dovresti andare a identificarli uno per uno in mezzo a quelli altrui — il log diventa un grafo ingarbugliato e l'operazione si complica.

Se vuoi che il tuo branch contenga solo i tuoi commit nella storia, la risposta è git rebase origin/dev invece di git merge origin/dev: i tuoi commit vengono riscritti sopra quelli del collega, la storia rimane una linea pulita e il cherry-pick futuro è banale. È esattamente per questo che esiste il Modulo 6.

Se il cherry-pick è una priorità — usa sempre il rebase

Con il merge risolvi i conflitti una volta sola, ma il risultato è un grafo ingarbugliato: il cherry-pick futuro resta scomodo indipendentemente da quanto bene hai risolto il conflitto. (→ Modulo 6: Il problema che risolve il rebase)

Con il rebase potresti dover risolvere un conflitto per ogni tuo commit riapplicato — è più laborioso sul momento. Ma il risultato è una storia lineare con i tuoi commit in cima, identificabili e cherry-pickabili con precisione chirurgica.

Il rebase è il prezzo che paghi ora per avere libertà operativa dopo. Se sai già che farai cherry-pick, non c'è discussione: sincronizza sempre con git rebase origin/dev.

Branch a vita breve. Un branch che dura settimane accumula divergenze enormi. Feature piccole, PR veloci: il ciclo di vita ideale di un branch è giorni, non settimane.

Riesumazione di un branch già mergiato

Un branch mergiato non sparisce finché non lo cancelli esplicitamente. Puoi tornare su quel branch, aggiungere commit e usarli per un cherry-pick futuro:

git switch feature/login
# fai le modifiche
git commit -m "fix: aggiunta validazione email"
# il nuovo commit è pulito e cherry-pickabile

Se il branch è già stato cancellato, puoi recuperarlo dall'hash dell'ultimo commit:

# ricrea il branch da un hash specifico
git branch feature/login a3f5c2d

# se non ricordi l'hash, il reflog ce l'ha (→ Modulo 7: git reflog)
git reflog | grep "feature/login"
Pro

✓ Semplicità — riprendi dove avevi lasciato con un semplice git switch

✓ Continuità di contesto — tutta la storia della feature è lì, utile per capire le decisioni precedenti

✓ Cherry-pick facile — i nuovi commit sono isolati e facilmente identificabili

Contro

✗ PR sporca — includerà anche i commit vecchi già mergiati, confondendo i reviewer

✗ Rischio duplicazione — i commit vecchi esistono già su dev, la storia può diventare ambigua

✗ Branch pesante — più dura, più è difficile tenerlo in sincronia con dev

Quando usare quale approccio

Riesuma il branch originale quando l'aggiunta è strettamente correlata alla feature e non devi aprire una PR — ti serve solo produrre commit puliti da cherry-pickare. Crea un branch nuovo quando devi aprire una PR, quando l'aggiunta è concettualmente separata, o quando il branch originale era già stato cancellato — in quel caso tanto vale partire pulito.

Comunica sui file condivisi. Se sai che un collega sta lavorando sullo stesso file, sincronizzatevi prima — decidete chi fa cosa per evitare l'overlap.

Allinea il branch prima della PR. Prima di aprire una Pull Request, porta il tuo branch in linea con il branch di destinazione:

# Aggiorna il tuo branch con gli ultimi commit di dev
git fetch
git merge origin/dev    # oppure: git rebase origin/dev (→ Modulo 6)
git push

# Ora la PR non ha divergenze — il review è più pulito
Regola pratica

Risolvi sempre i conflitti sul tuo branch, non su dev o main. Porta le modifiche di dev dentro feature/x, risolvi lì, poi apri la PR. In questo modo se sbagli qualcosa non hai mai toccato il branch condiviso.

Rebase

Il problema che risolve il rebase

Nel Modulo 5 abbiamo visto che sincronizzare il tuo branch con git merge origin/dev funziona, ma ha un costo: i commit dei colleghi entrano nella storia del tuo branch, il log diventa un grafo ingarbugliato, e se in futuro vuoi fare cherry-pick dei tuoi commit devi andare a cercarli uno per uno in mezzo a quelli altrui.

Il rebase risolve esattamente questo problema. Invece di creare un merge commit che unisce le due storie, ri-applica i tuoi commit sopra quelli del collega — come se avessi iniziato a lavorare dopo di lui, non in parallelo. Il risultato è una storia lineare, pulita, dove i tuoi commit sono sempre in cima.

Richiamo dal Modulo 5

Nel Modulo 5 abbiamo già visto il confronto merge vs rebase nell'ottica del cherry-pick: con il merge i tuoi commit si perdono in mezzo al grafo, con il rebase rimangono sempre in cima e sono identificabili con precisione. Qui approfondiamo come il rebase funziona davvero, e quando usarlo — o non usarlo.

Come funziona il rebase internamente

Questo è il punto che più spesso viene frainteso: il rebase non sposta i commit. Li ricrea. Ogni commit è immutabile in Git — ha un hash calcolato sul suo contenuto, sul suo parent, sull'autore e sul timestamp. Non puoi "spostarlo" su una nuova base senza cambiarne il parent, e cambiare il parent significa cambiare l'hash. Quindi Git crea commit nuovi, con lo stesso contenuto ma hash diversi.

Il processo passo per passo è questo:

1. Git trova il merge base — l'antenato comune tra il tuo branch e il branch di destinazione (il punto C nel diagramma sopra).

2. Salva da parte i diff dei tuoi commit dalla merge base in poi (E e F nel nostro esempio) — li chiama internamente patch.

3. Sposta HEAD al tip del branch di destinazione (D).

4. Ri-applica le patch una per una, creando nuovi commit (E' e F') con il parent aggiornato.

5. Sposta il puntatore del tuo branch sull'ultimo commit riscritto (F').

I vecchi commit non spariscono subito

I commit originali E e F esistono ancora nell'object store di Git finché il garbage collector non li pulisce (di solito dopo 30 giorni). Se ti accorgi subito di aver fatto un rebase sbagliato, git reflog te li mostra ancora e puoi tornare indietro. Ma non contarci dopo settimane. (→ Modulo 7: git reflog — la rete di sicurezza)

git rebase — uso base

Il caso d'uso più comune: sei su un feature branch e vuoi allinearlo con le ultime modifiche di dev prima di aprire la PR.

# 1. Assicurati di essere sul tuo branch
git switch feature/login

# 2. Scarica gli aggiornamenti dal remote (senza applicarli ancora)
git fetch

# 3. Ri-applica i tuoi commit sopra l'ultimo dev
git rebase origin/dev

# 4. Aggiorna il feature branch sul remote con la storia riscritta
#    --force-with-lease è necessario perché gli hash sono cambiati — il remote aveva
#    i vecchi commit (E, F), tu stai pushando quelli riscritti (E', F').
#    Senza --force-with-lease Git rifiuterebbe il push perché le storie non sono
#    compatibili in modo fast-forward.
git push --force-with-lease

Dopo il rebase, il tuo branch è esattamente come se avessi iniziato a lavorare dall'ultimo commit di dev. La PR sarà pulita, senza grafo, e Azure DevOps mostrerà solo i tuoi commit. In più, E' e F' sono cherry-pickabili con precisione: li trovi subito in cima alla storia, con hash puliti e nessun commit altrui in mezzo — se in futuro dovessi portare solo uno dei tuoi commit su un altro branch, sai esattamente dove andare. Con il merge invece i tuoi commit si trovano sepolti nel grafo, e identificarli richiede più lavoro.

Anatomia — git rebase origin/dev

ParteSignificato
git rebaseri-applica i commit del branch corrente sopra un nuovo punto base
origin/devil nuovo punto base — l'ultimo commit di dev sul remote. Puoi usare anche un branch locale (dev) o un hash diretto
Perché --force-with-lease e non --force

git push --force sovrascrive il remote senza controlli — se nel frattempo un collega ha pushato qualcosa sul tuo branch, lo perdi senza accorgertene. --force-with-lease controlla prima che il remote sia esattamente dove ti aspetti: se qualcuno ha pushato nel frattempo, Git ti blocca con un errore e ti avvisa. È la versione sicura del force push — usala sempre.

Quando --force-with-lease ti blocca, non fare nulla in modo impulsivo. Prima di tutto guarda cosa ha pushato il collega:

git fetch
git log origin/feature/login --oneline

A quel punto hai tre scenari:

I commit del collega sono da tenere e vuoi un mix dei due contributi — i commit del collega sono su origin/feature/login, non su origin/dev, quindi non basta il rebase che hai già fatto. Servono due rebase su sorgenti diverse:

git fetch
git rebase origin/dev              # allinei con dev (come sempre)
git push --force-with-lease        # → bloccato: il collega ha pushato su feature/login nel frattempo
git rebase origin/feature/login    # integri i commit del collega sul tuo branch
git push --force-with-lease        # ora passa

Il primo rebase porta le novità di dev. Il secondo porta i commit del collega da feature/login — sono due sorgenti diverse, non puoi fare tutto in uno. Se ci sono sovrapposizioni sugli stessi file, Git ti ferma sui conflitti da risolvere uno per uno.

I commit del collega sono da scartare (push per errore) — parla prima con il collega e accertatevi di essere d'accordo. Solo dopo:

git push --force-with-lease  # sovrascrive i commit del collega

Non sei sicuro di cosa fare — non toccare niente e parla con il collega. --force-with-lease ti ha fermato apposta per farti ragionare: il guard rail tecnico funziona, ma la comunicazione è la vera soluzione.

Conflitti durante il rebase

Anche il rebase può incontrare conflitti — ma funziona in modo diverso dal merge. Con il merge hai un unico punto di conflitto alla fine. Con il rebase, ogni tuo commit viene ri-applicato uno alla volta: se hai 4 commit e il secondo genera un conflitto, Git si ferma lì, ti chiede di risolverlo, e poi continua con il terzo e il quarto.

Un punto importante da capire: il numero di possibili conflitti dipende dai tuoi commit, non da quelli del collega. Se il collega ha fatto 3 commit e tu ne hai 4, potresti avere fino a 4 punti di conflitto — uno per ogni tuo commit ri-applicato. I commit del collega influenzano solo dove si genera il conflitto: se lui ha toccato gli stessi file che tocchi tu in qualche tuo commit, lì Git si ferma.

Esempio: collega ha fatto D1, D2, D3 — tuoi commit da ri-applicare sono E, F, G

  D1 → modifica AuthService.cs
  D2 → modifica Program.cs
  D3 → modifica appsettings.json

  E → modifica AuthService.cs   ← conflitto con D1 (stesso file)
  F → modifica Logger.cs        ← nessun conflitto (file diverso)
  G → modifica Program.cs       ← conflitto con D2 (stesso file)

Risultato: fino a 2 conflitti (su E e su G) — non 3, perché D3 tocca un file che nessuno dei tuoi commit tocca.

In pratica, se sincronizzi spesso e i commit sono atomici, i conflitti durante un rebase sono rari e localizzati.

Il workflow quando il rebase si ferma

# Rebase in corso — Git si ferma su un conflitto
git rebase origin/dev
# output: CONFLICT (content): Merge conflict in src/Services/AuthService.cs
# output: error: could not apply a3f5c2d... feat: aggiunge validazione email

# 1. Risolvi il conflitto — hai tre strade:

#    STRADA A — risolvi a mano (sempre sicura)
#    Apri il file, scegli cosa tenere tra i marker <<<< ==== >>>> e salva

#    STRADA B — vuoi tenere interamente LA TUA versione del file
#    Attenzione: durante il rebase --theirs = il TUO commit (logica invertita!)
git checkout --theirs src/Services/AuthService.cs

#    STRADA C — vuoi tenere interamente la versione di origin/dev
#    Attenzione: durante il rebase --ours = origin/dev (logica invertita!)
git checkout --ours src/Services/AuthService.cs

# 2. Segna i file come risolti
git add src/Services/AuthService.cs

# 3. Continua il rebase — Git ri-applica il prossimo commit
git rebase --continue

# Se vuoi annullare tutto e tornare allo stato precedente
git rebase --abort
--ours e --theirs si invertono durante il rebase (← confronta con il merge nel Modulo 5)

Durante un rebase Git sposta temporaneamente HEAD sul branch di destinazione (origin/dev) e ri-applica i tuoi commit uno alla volta sopra. Dal punto di vista di Git in quel momento:

--ours = origin/dev (il branch del collega) — perché Git "è su" origin/dev
--theirs = il tuo commit che sta venendo ri-applicato

È l'opposto esatto di quello che ci si aspetta. Per questo le strade B e C vanno usate solo quando sei sicuro al 100% di voler scartare interamente una delle due versioni — con un solo comando butti via tutto il lavoro su quel file senza poter rivedere riga per riga. In caso di dubbio, usa sempre la strada A.

Rebase interattivo — git rebase -i

Il rebase interattivo è uno degli strumenti più potenti di Git. Ti permette di riscrivere la storia dei tuoi commit locali prima di pusharli: puoi cambiarne l'ordine, unirli, dividerne il messaggio, eliminarne uno, correggere una svista. È il modo di tenere la storia del branch pulita e professionale.

Si lancia così:

# Apri l'editor interattivo per modificare gli ultimi N commit
git rebase -i HEAD~3   # modifica gli ultimi 3 commit

# oppure a partire da un commit specifico (escluso)
git rebase -i a3f5c2d

Salvi e chiudi. Git esegue le istruzioni dall'alto in basso:

1. pick c2a8f30 → tiene il commit AuthService così com'è

2. fixup b7d1e9f → prende le modifiche del wip e le incolla dentro AuthService — il commit wip sparisce

3. pick a3f5c2d → tiene il commit validazione così com'è

Risultato — da 3 commit sei passato a 2, il wip è sparito dalla storia:

a1b2c3d feat: aggiunge struttura AuthService   # contiene anche le modifiche del wip
e4f5g6h feat: aggiunge validazione email
Squash e cherry-pick non vanno d'accordo

Quando schiacci più commit in uno con squash o fixup, il risultato è un singolo commit che contiene tutte le modifiche insieme. Se in futuro dovessi fare cherry-pick, puoi portare solo quel commit intero — non puoi separare le modifiche originali che hai compresso. Esempio: se hai squashato "aggiunge struttura AuthService" e "aggiunge validazione email" in un unico commit, non puoi più portare solo la validazione su un altro branch senza portarti dietro anche la struttura. Decide bene prima di fare squash: se le modifiche potrebbero servire separatamente in futuro, tienile in commit distinti.

Puoi anche riordinare i commit

Nel file del rebase interattivo puoi spostare fisicamente le righe per cambiare l'ordine dei commit. Git li ri-applica nell'ordine in cui li trovi — dall'alto in basso. Attenzione: se due commit toccano le stesse righe, riordinarli può creare conflitti.

Reword — correggere solo il messaggio

# Vuoi solo correggere il messaggio dell'ultimo commit
git rebase -i HEAD~1

# Nel file, cambia pick in reword:
reword a3f5c2d feat: aggiunge validazione emial   # typo nel messaggio

# Salva e chiudi — Git riaprì l'editor con il messaggio del commit per correggere il typo
Alternativa rapida per l'ultimo commit

Se devi correggere solo l'ultimo commit (messaggio o contenuto), git commit --amend è più veloce del rebase interattivo — non apre la lista, modifica direttamente l'ultimo commit. (→ Modulo 8: git commit --amend in dettaglio)

git pull --rebase

Nel Modulo 4 abbiamo già incontrato git pull --rebase come alternativa a git pull. Ora che sai come funziona il rebase, è chiaro cosa fa: invece di scaricare i nuovi commit e creare un merge commit, li scarica e ci ri-applica sopra i tuoi commit locali.

La differenza rispetto a un git rebase diretto è che git pull --rebase fa prima il fetch automaticamente — sono equivalenti:

# Questi due blocchi fanno la stessa cosa:
git fetch
git rebase origin/dev

# è identico a:
git pull --rebase origin dev

La differenza pratica è di sicurezza: con git rebase origin/dev diretto stai lavorando su quello che Git ha in locale di origin/dev — se non hai fetchato di recente, potresti fare il rebase su una base già vecchia e accorgertene solo al push quando --force-with-lease ti blocca. git pull --rebase ti evita di dimenticarti il fetch.

# git pull standard — scarica + merge commit se ci sono commit locali
git pull

# git pull --rebase — scarica + rebase, storia lineare
# forma esplicita: specifica da dove pullare (consigliata per chiarezza)
git pull --rebase origin dev

# forma breve: funziona solo se il tracking branch è già configurato
# (lo è automaticamente dopo il primo git push -u origin <branch>)
git pull --rebase

# Se il tracking branch NON è configurato, git pull --rebase dà questo errore:
# There is no tracking information for the current branch.
# Please specify which branch you want to rebase against.
# → soluzione 1: usa la forma esplicita
git pull --rebase origin dev
# → soluzione 2: configura il tracking branch e poi usa la forma breve
git branch --set-upstream-to=origin/dev feature/login
git pull --rebase

# Puoi impostarlo come comportamento default
git config --global pull.rebase true

In un team che lavora su branch shared (tipo dev), git pull --rebase è spesso preferito perché evita i merge commit inutili nei pull quotidiani — quei "Merge branch 'dev' of https://..." che non aggiungono nulla alla storia.

Attenzione se hai commit locali non ancora pushati

git pull --rebase riscrive i tuoi commit locali (cambia i loro hash). Se li hai già pushati su un branch condiviso, avrai un problema al prossimo push — vedi la sezione "Regola d'oro" qui sotto.

La regola d'oro del rebase

Il rebase è potente ma ha una regola fondamentale che non si può violare:

⚠ Regola d'oro

Non fare mai rebase di commit che hai già pushato su un branch condiviso — cioè un branch su cui lavorano anche altri (tipicamente dev, main, o qualsiasi branch aperto in PR da altri).

Il motivo è semplice: quando fai rebase, i commit vengono riscritti con nuovi hash. Se i tuoi colleghi avevano già scaricato i commit originali sul loro repo locale, ora hanno una storia che diverge da quella che hai pushato. Il risultato è caos: Git non riesce a riconciliare le due storie senza che ogni collega faccia operazioni manuali, e la storia del branch diventa un disastro con commit duplicati.

Perché il ? è un disastro — spiegazione

Il collega ha ancora in locale A → B → C con i vecchi hash. Quando fa git pull, Git confronta la sua storia con quella del remote — che ora finisce su C' (hash diverso). Non è una semplice situazione di "sei indietro di N commit": sono due storie strutturalmente incompatibili. Git non riesce a fare un fast-forward perché i commit B e C del collega non esistono più sul remote — sono stati sostituiti da B' e C' che sono oggetti completamente diversi.

Git si rifiuta di procedere e mostra questo errore:

hint: You have divergent branches and need to specify how to reconcile them.
hint:   git config pull.rebase false  # merge
hint:   git config pull.rebase true   # rebase
hint:   git config pull.ff only       # fast-forward only
fatal: Need to specify how to reconcile divergent branches.

Il segnale di allarme è "divergent branches" — in condizioni normali un git pull non dovrebbe mai dare questo errore. Il collega capisce che qualcosa non va, ma senza sapere che hai fatto un force push non sa cosa è successo.

Cosa può fare il collega a questo punto:

Opzione 1 — ha solo A → B → C, nessun commit aggiuntivo
La sua storia locale coincide esattamente con quella che hai riscritto tu — solo gli hash sono diversi. Può allinearsi al remote scartando i suoi B e C e prendendo B' e C':

git fetch
git reset --hard origin/feature/login

⚠ --hard è distruttivo: le modifiche contenute in B e C spariscono dal working tree — non finiscono in nessun posto sicuro. L'unica rete di sicurezza è il reflog: Git tiene traccia di tutti i movimenti di HEAD, quindi B e C esistono ancora nell'object store per circa 30 giorni prima che il garbage collector li pulisca. Se il collega si accorge subito di aver perso qualcosa può recuperarli. (→ Modulo 7: git reset · git reflog)

git reflog                  # trova l'hash di B o C
git checkout <hash>         # recupera il commit

Ma è una manovra di emergenza. L'Opzione 1 va usata solo se il collega è assolutamente sicuro di non avere lavoro aggiuntivo su B e C. Se c'è anche il minimo dubbio, meglio l'Opzione 2.

Opzione 2 — ha A → B → C più commit aggiuntivi suoi (es. E, F)
Il collega ha lavorato sopra B e C aggiungendo commit suoi che non vuole perdere. Non può buttare via tutto — deve ri-applicare solo i suoi commit aggiuntivi (E e F) sopra la storia riscritta:

git fetch
git rebase origin/feature/login

Git ri-applica E e F sopra C', producendo A → D → B' → C' → E' → F'. Se E o F toccano le stesse righe di B' o C', ci saranno conflitti da risolvere uno per uno.

Opzione 3 — sceglie merge senza capire cosa è successo
Tecnicamente funziona ma genera una storia disastrosa con B, C, B' e C' tutti presenti — commit duplicati con contenuto identico ma hash diversi. Da evitare assolutamente.

In entrambi i casi il collega deve sapere cosa è successo per scegliere l'opzione giusta — da qui l'importanza di comunicare prima di fare un force push su un branch condiviso. Moltiplicato per ogni collega che aveva scaricato il branch, diventa un problema serio di team.

Il rebase è sicuro finché i commit che stai riscrivendo non sono mai stati pushati, oppure sono su un branch che usi solo tu e nessun altro ha clonato o scaricato.

SituazioneRebase sicuro?
Commit locali mai pushati✓ Sempre sicuro — nessun altro ha quegli hash
Feature branch privato (solo tuo)✓ Sicuro — sei l'unico che lo usa. Servirà --force-with-lease al push
Feature branch su cui lavorano altri✗ Pericoloso — comunica prima con il team
Branch condivisi (dev, main)✗ Mai — è la violazione più grave della regola d'oro
Se hai già pushato su un branch condiviso — il trade-off

Una volta pushato su un branch che altri hanno già scaricato, non puoi più fare rebase senza rompere la storia per i colleghi. L'unica alternativa sicura per integrare le novità di dev è git merge origin/dev:

git fetch
git merge origin/dev
git push   # nessun force push — la storia non è stata riscritta

Ma c'è un costo: i commit di dev entrano nella storia del tuo branch, il log diventa un grafo, e il cherry-pick futuro si complica — esattamente come abbiamo visto nel Modulo 5 → Squash merge. Non esiste una soluzione tecnica perfetta una volta che sei in questa situazione:

SceltaProContro

git rebase + force pushstoria lineare, cherry-pick facilerompe la storia per chiunque abbia già scaricato il branch

La soluzione vera è di processo, non tecnica: il feature branch dovrebbe essere privato per default finché non apri la PR — solo tu ci lavori, solo tu ci pusha, e puoi fare rebase liberamente. Quando apri la PR e altri iniziano a lavorarci o a reviewarla, smetti di fare rebase. Decidere prima se il branch è privato o condiviso ti evita di trovarti in questo trade-off.

Rebase vs merge — quando scegliere cosa

Non esiste una risposta universale: dipende dal team, dal tipo di progetto e da cosa vuoi ottenere dalla storia. Ecco le linee guida pratiche:

Il pattern che funziona in quasi tutti i team

Usa il rebase sul tuo feature branch per tenerlo allineato con dev e per pulire i commit prima della PR. Usa il merge (o squash merge via PR) per integrare la feature completata in dev. Non toccare mai rebase sui branch condivisi.

In ottica cherry-pick: lavora sempre su feature branch

Se sai che farai cherry-pick, la regola è semplice: ogni modifica, anche piccola, su un feature branch dedicato — mai direttamente su dev o main.

Il motivo è esattamente quello che abbiamo visto: su un FB privato puoi fare rebase liberamente e mantenere una storia lineare con i commit identificabili con precisione. Su dev e main non puoi mai fare rebase — prima o poi ci finisce dentro il merge commit di qualcun altro e il cherry-pick diventa scomodo o impossibile.

In Azure DevOps questo è già imposto dal workflow: dev e main sono branch protetti, nessuno ci pusha direttamente. È un guard rail di processo che ti mette automaticamente nella condizione giusta per il cherry-pick.

Modulo 7 · Avanzato

Reset, revert, restore

Quando qualcosa va storto — e prima o poi va storto — questi tre comandi sono la tua cassetta degli attrezzi per tornare indietro nel tempo senza panico.

7.1 — Il problema: "voglio annullare qualcosa"

Hai committato troppo presto? Hai scritto un messaggio di commit orribile? Hai pushato qualcosa che non dovevi? Prima di lanciare comandi a caso, fermati un secondo e chiediti tre cose:

🧭 Le 3 domande prima di annullare

1. Cosa voglio annullare? Un commit? Una modifica non committata? L'intero contenuto di un file?

2. Il commit è già stato pushato? Se sì, la strategia cambia completamente — non puoi semplicemente riscrivere la storia su un branch condiviso.

3. Voglio conservare le modifiche nel working tree, o buttarle via definitivamente? La differenza tra --soft, --mixed e --hard sta tutta qui.

In base alle risposte userai uno di questi tre strumenti:

    VUOI ANNULLARE...
    ├── modifiche non ancora in staging ──────────────→  git restore
    ├── file già in staging (ma non committato) ───────→  git restore --staged
    ├── commit locali (non ancora pushati) ────────────→  git reset
    └── commit già pushati (branch condiviso) ─────────→  git revert

7.2 — git restore: annulla modifiche non committate

git restore è il comando più "safe" di tutti: agisce solo sul working tree o sull'area di staging, non tocca mai la storia dei commit. Perfetto quando hai modificato un file e vuoi tornare all'ultima versione committata.

Caso 1 — Hai modificato un file ma non hai ancora fatto git add

# Situazione: hai modificato Program.cs ma vuoi annullare
git restore Program.cs

# Per annullare TUTTE le modifiche nel working tree in una botta
git restore .
⚠️ Attenzione: irreversibile sul working tree

git restore butta via le modifiche non staged per sempre — non finiscono nel cestino, non c'è undo. Usalo solo se sei sicuro di non voler conservare quelle modifiche.

ParteSignificato
giteseguibile Git
restoreripristina il contenuto di file nel working tree o in staging
Program.csil file da ripristinare; usa . per tutti i file

Caso 2 — Hai già fatto git add (file in staging) ma non hai ancora committato

# Togli il file dall'area di staging (le modifiche restano nel file)
git restore --staged Program.cs

# Togli TUTTO dall'area di staging
git restore --staged .

Questo è l'equivalente del vecchio git reset HEAD <file> — il file torna "modified" ma non staged. Le tue modifiche nel file rimangono intatte.

Caso 3 — Vuoi recuperare la versione di un file da un commit specifico

# Ripristina Program.cs com'era 3 commit fa
git restore --source HEAD~3 Program.cs

# O da un commit specifico tramite hash
git restore --source a3f8c21 Program.cs

Il file viene ripristinato nel working tree ma non viene committato automaticamente: puoi ispezionarlo, e poi decidere se fare add + commit o scartarlo di nuovo.

✅ Vuoi tornare alla versione attuale dopo un --source?

HEAD non si è mosso — il file è cambiato solo nel working tree. Per ripristinare la versione dell'ultimo commit basta:

git restore Program.cs

Senza --source, il default è già HEAD.

7.3 — git reset: sposta HEAD indietro

git reset sposta il puntatore HEAD (e il branch corrente) a un commit precedente. È lo strumento giusto per annullare commit locali — cioè commit che non hai ancora pushato.

🚨 Regola d'oro: mai reset su commit già pushati in un branch condiviso

Se il commit è già su Azure DevOps e altri sviluppatori lo hanno pullato, un reset seguito da git push --force riscrive la storia pubblica e crea danni al team. Per i commit già pushati usa git revert. (→ sezione 7.4: git revert)

Esistono tre modalità di reset, che differiscono per cosa succede alle modifiche del commit che stai annullando:

                          STAGING    WORKING TREE
git reset --soft  <target>  intatto    intatto       ← commit annullato, modifiche pronte per re-committare
git reset --mixed <target>  svuotato   intatto       ← default — modifiche nel file ma da ri-aggiungere
git reset --hard  <target>  svuotato   CANCELLATO    ← tutto sparito, working tree identico al target commit

⚠️  <target> è OBBLIGATORIO — es. HEAD~1, HEAD~3, oppure un hash come a3f8c21

--soft: "annulla il commit, tieni tutto in staging"

# Annulla l'ultimo commit — le modifiche tornano in staging, pronte per un nuovo commit
git reset --soft HEAD~1

Scenario tipico: hai committato troppo presto, vuoi aggiungere altri file prima di rifare il commit con un messaggio migliore.

ParteSignificato
git resetsposta HEAD (e il branch) al commit indicato
--softconserva le modifiche in staging area
HEAD~1un commit prima dell'attuale; usa HEAD~3 per tornare di 3, o un hash specifico

--mixed: "annulla commit e staging, tieni le modifiche nei file" (default)

# Annulla l'ultimo commit — le modifiche restano nei file ma escono dallo staging
git reset HEAD~1
# equivalente a: git reset --mixed HEAD~1

È il comportamento default quando non specifici nulla. Le modifiche restano nei tuoi file, ma devi rifare git add prima di committare.

--hard: "annulla tutto, voglio tornare esattamente a quel commit"

# ⚠️ DISTRUTTIVO — cancella anche le modifiche nel working tree
git reset --hard HEAD~1

# Torna all'ultimo commit del remote (butta via tutto il lavoro locale non pushato)
git reset --hard origin/main
⚠️ --hard è irreversibile (quasi)

Le modifiche nel working tree vengono cancellate definitivamente. L'unica ancora di salvezza è il reflog — i commit rimangono nell'oggetto store di Git per un po' di tempo prima di essere rimossi dal garbage collector. (→ sezione 7.5: git reflog)

7.4 — git revert: annulla un commit senza riscrivere la storia

Quando un commit è già stato pushato su un branch condiviso (come main o dev), non puoi usare reset. La soluzione è git revert: crea un nuovo commit che fa l'esatto contrario delle modifiche del commit che vuoi annullare. La storia rimane intatta, e il tuo revert è visibile a tutti.

# Annulla l'ultimo commit (crea un nuovo "revert commit")
git revert HEAD

# Annulla un commit specifico tramite hash
git revert a3f8c21

# Annulla senza aprire l'editor per il messaggio (usa il messaggio di default)
git revert HEAD --no-edit

# Dopo il revert, pusha normalmente
git push
ParteSignificato
git revertcrea un nuovo commit che inverte le modifiche del commit indicato
HEADil commit da annullare; puoi usare un hash come a3f8c21
--no-editnon apre l'editor — usa il messaggio di default "Revert: ..."
💡 Cosa fa --no-edit?

Senza --no-edit, Git apre automaticamente l'editor di testo (Vim o quello configurato) con un messaggio precompilato tipo Revert "fix: corretto calcolo IVA" e aspetta che tu lo confermi o modifichi. Con --no-edit salta questo passaggio e usa il messaggio di default direttamente — più veloce quando il messaggio automatico va bene così com'è.

💡 reset vs revert: la differenza chiave

reset → riscrive la storia (usa solo su commit locali, mai su branch condivisi)

revert → aggiunge un commit che annulla (sicuro su branch condivisi, storia sempre tracciabile)

Nel contesto di un team su Azure DevOps, revert è quasi sempre la scelta giusta per annullare qualcosa già pushato.

✅ Cosa vede il collega quando fa git pull?

Per lui è un pull normale. Arriva un nuovo commit con messaggio tipo Revert "fix: corretto calcolo IVA" e il codice è già sistemato — non deve fare nulla di speciale, nessun conflitto. L'unica cosa visibile è quel commit di revert in git log, ed è giusto così: è la traccia di quello che è successo. Trasparenza totale.

Revert di più commit in sequenza

# Annulla un range di commit (dal più recente al più vecchio — ordine importante!)
git revert HEAD~3..HEAD --no-edit

# Oppure uno alla volta — più controllo, stesso risultato
git revert HEAD --no-edit
git revert HEAD~1 --no-edit
git revert HEAD~2 --no-edit

7.5 — git reflog: la rete di sicurezza definitiva

Il reflog è la cosa più importante che probabilmente non sapevi di avere. È un registro locale di tutti i movimenti di HEAD — ogni commit, checkout, reset, rebase che hai fatto — anche quelli che sembrano "cancellati". Git conserva questo registro per 90 giorni per default.

⚠️ Il reflog esiste solo sul tuo PC

Il reflog è esclusivamente locale — non viene pushato su Azure DevOps e non è condiviso con nessuno. Ogni sviluppatore ha il proprio. Conseguenza pratica: se perdi la macchina, cancelli il repository locale, o un collega ha fatto un casino sul suo repo, il reflog non ti aiuta. È una rete di sicurezza personale, non di team.

# Mostra il reflog di HEAD
git reflog

# Output tipico:
a3f8c21 HEAD@{0}  reset: moving to HEAD~1
7b2e934 HEAD@{1}  commit: fix: corretto calcolo IVA
f91a3bc HEAD@{2}  commit: feat: aggiunto metodo Pagamento
c44d210 HEAD@{3}  checkout: moving from main to feature/pagamenti

Ogni riga ha il formato hash HEAD@{N}. HEAD@{0} è lo stato attuale, HEAD@{1} è quello precedente, e così via.

Scenario pratico: hai fatto un reset --hard per errore

# 1. Trova l'hash del commit che volevi conservare
git reflog
#    → vedi che HEAD@{1} era il tuo commit "fix: corretto calcolo IVA" con hash 7b2e934

# 2. Torna a quello stato con un reset --hard
git reset --hard HEAD@{1}
# oppure con l'hash diretto
git reset --hard 7b2e934
✅ Il reflog ti salva anche dopo un rebase andato male

Come visto nel Modulo 6, dopo un rebase che ha prodotto risultati inaspettati puoi usare il reflog per tornare allo stato pre-rebase. Cerca la riga con rebase (start) e resetta all'hash immediatamente precedente.

# Esempio: tornare allo stato pre-rebase
git reflog
# a3f8c21 HEAD@{0}  rebase (finish): returning to refs/heads/feature/pagamenti
# 7b2e934 HEAD@{1}  rebase (start)
# f91a3bc HEAD@{2}  commit: feat: aggiunto metodo Pagamento  ← qui eri prima del rebase

git reset --hard HEAD@{2}

7.6 — Scenario completo: workflow di emergenza

Metti tutto insieme con un caso reale. Immagina questa situazione:

📋 Scenario

Hai committato e pushato su main un file di configurazione con credenziali hardcodate. Devi annullarlo subito senza rompere la storia del branch condiviso.

# 1. Identifica il commit da annullare
git log --oneline -5
# a3f8c21 (HEAD -> main, origin/main)  config: aggiornato appsettings  ← questo è il colpevole
# 7b2e934  feat: aggiunto metodo Pagamento

# 2. Usa revert (non reset! il commit è già pushato)
git revert a3f8c21 --no-edit

# 3. Pusha il revert commit

7.8 — Impatto sul cherry-pick

I comandi di questo modulo possono influire sul cherry-pick, ma solo in alcuni casi specifici. La regola generale: finché non riscrivi la storia pubblica, il cherry-pick non ne risente.

ComandoImpatto sul cherry-pick
git restore✅ Nessun impatto — agisce solo sul working tree, non tocca la storia
git revert✅ Nessun impatto — aggiunge un commit, storia intatta, cherry-pick funziona normalmente
git reset (solo locale, non pushato)✅ Nessun impatto finché non pushato
git reset + force push su branch condiviso⚠️ Pericoloso — gli hash dei commit cambiano. Se qualcuno aveva già fatto cherry-pick di quei commit, si ritrova commit duplicati o conflitti
⚠️ Il vero nemico del cherry-pick non è il reset

Il problema non è il reset in sé, ma il force push su branch condivisi che può derivarne. Riscrivere la storia pubblica cambia gli hash dei commit — e il cherry-pick lavora proprio sugli hash. Tutto quello che non tocca la storia pubblica è sicuro.

8.1 — git stash: mettere da parte il lavoro

Scenario classico: stai lavorando su un feature branch, hai modifiche a metà, e arriva una notifica — bug critico in produzione, devi switchare subito su main. Non puoi committare roba incompleta. Qui entra git stash.

git stash salva temporaneamente lo stato del working tree e dello staging area in una struttura a stack separata, riportando i tuoi file allo stato dell'ultimo commit — come se non avessi toccato niente. Poi puoi switchiare branch, fixare, tornare, e riestrarre il lavoro.

Il flusso base

# Salva tutto (working tree + staging) nello stash
git stash

# Switcha al branch del bug fix
git checkout main
# ... fix, commit, push ...

# Torna al tuo branch
git checkout feature/mia-feature

# Riestrái il lavoro dallo stash
git stash pop

git stash listmostra tutti gli stash salvati con indice e messaggio
git stash drop stash@{N}cancella lo stash numero N senza applicarlo

Se usi spesso lo stash ti ritrovi con un elenco di WIP on feature/x: abc123 ... indistinguibili. Dai sempre un messaggio:

Lo stash è pensato per interruzioni temporanee e brevi, non per tenere lavoro "in pausa" per giorni. Se stai mettendo da parte roba per più di qualche ora, considera di committare su un branch con un messaggio WIP: ... e fare un git reset --soft HEAD~1 quando riprendi — è molto più sicuro e tracciabile. 8.2 — git tag: etichettare le release

Un tag in Git è un puntatore con un nome leggibile che punta a un commit specifico. A differenza di un branch, non si sposta mai — rimane ancorato a quel commit per sempre. È lo strumento standard per marcare le release: v1.0.0, v2.3.1, ecc.

Tag lightweight Semplicemente un puntatore a un commit, come un alias. Nessuna informazione extra. Non consigliato per release ufficiali. Tag annotato Un oggetto Git vero e proprio con: autore, data, messaggio, firma GPG opzionale. È il tipo corretto per marcare versioni di produzione.

Creare un tag annotato (il modo giusto per le release)

# Assicurati di essere sul commit che vuoi taggare
git checkout main
git pull

# Crea il tag annotato
git tag -a v1.0.0 -m "Release 1.0.0 — prima versione stabile"

# Pusha il tag su Azure DevOps
git push origin v1.0.0

8.3 — git commit --amend: correggere l'ultimo commit

git commit --amend ti permette di riscrivere l'ultimo commit — sia il messaggio sia il contenuto. È l'operazione che usi quando fai commit e ti accorgi subito dopo di un typo nel messaggio, di un file dimenticato, o di una modifica che andava inclusa.

⚠️ Amend riscrive la storia — usalo solo su commit non ancora pushati

8.5 — git bisect: trovare il commit del bug con bisezione binaria

Hai una build che funzionava 200 commit fa e ora è rotta. Controllare commit per commit sarebbe un'agonia. git bisect risolve questo con un algoritmo di ricerca binaria: divide la storia a metà, ti porta lì, tu dici se il bug c'è o no, e Git divide ancora. Con 200 commit bastano circa 8 passi per trovare il commit esatto.

Il flusso base di bisect

# 1. Avvia la sessione di bisect
git bisect start

# 2. Indica il commit buono (funzionava)
git bisect good v1.2.0         # puoi usare un tag, un hash, o HEAD~N

# 3. Indica il commit cattivo (è rotto)
git bisect bad HEAD            # di solito il punto corrente

# Git ti porta automaticamente a un commit intermedio
# Testa: compila, esegui, controlla se il bug c'è

# 4. Se il bug C'È in questo commit:
git bisect bad

# 4. Se il bug NON C'È in questo commit:
git bisect good

# Ripeti il passo 4 finché Git non ti dice qual è il commit colpevole
# Output finale: "abc1234 is the first bad commit"

# 5. Quando hai finito, torna allo stato normale
git bisect reset
ParteSignificato
git bisect startavvia la sessione — Git entra in modalità bisect
git bisect good <ref>marca questo punto come "funzionante" — definisce il confine inferiore della ricerca
git bisect bad <ref>marca questo punto come "rotto" — definisce il confine superiore; senza argomento usa HEAD
git bisect good / baddurante la ricerca, classifica il commit corrente in cui Git ti ha portato
git bisect resettermina la sessione e riporta HEAD al branch di partenza — sempre da eseguire alla fine
git bisect logmostra la storia della sessione bisect in corso — utile per capire dove sei
git bisect skipsalta il commit corrente (es. non compila, non puoi testarlo)
ℹ️ Quanti passi servono?

La ricerca binaria su N commit richiede al massimo log₂(N) passi. Con 100 commit bastano 7 passi, con 1000 commit bastano 10 passi. È uno degli strumenti più potenti di Git per il debugging — e quasi nessuno lo usa, il che ti mette in vantaggio.

Bisect automatico con uno script

Se hai un test automatico che ritorna 0 quando funziona e 1 quando è rotto, puoi fare fare tutto il lavoro a Git:

git bisect start

git bisect good v1.2.0

# Con Gradle:

Durante una sessione bisect sei in detached HEAD — Git ti sposta di commit in commit fuori da qualsiasi branch. Se dimentichi git bisect reset alla fine, rimani in questo stato e i commit successivi non appartengono a nessun branch. Sempre, sempre, sempre: git bisect reset alla fine. (→ Modulo 3: detached HEAD)

Stesso discorso vale per cherry-pick: non eseguirlo mai durante una sessione bisect attiva — il commit finirebbe su HEAD senza appartenere a nessun branch e sembrerebbe sparire al momento del git bisect reset. (→ Modulo 9: cherry-pick best practice)

8.6 — git worktree: due branch, una sola cartella

Normalmente puoi lavorare su un solo branch alla volta perché il tuo working tree è unico. git worktree cambia questo: ti permette di avere più working tree dello stesso repository su disco contemporaneamente, ognuno su un branch diverso, senza clonare niente.

Caso d'uso tipico: stai sviluppando su feature/X e devi fare un hotfix su main senza interrompere il tuo lavoro — e senza stash, e senza perdere il contesto. (→ alternativa con git stash — sezione 8.1)

# Aggiunge un secondo working tree in una cartella separata, su branch main
git worktree add ../repo-hotfix main

# Ora hai due cartelle: la tua normale e ../repo-hotfix (su main)
# Vai nella nuova cartella, fai il fix, commit, push — come normale
cd ../repo-hotfix
# ... lavora ...

# Quando hai finito, rimuovi il worktree aggiuntivo
git worktree remove ../repo-hotfix

La differenza rispetto a clonare il repo in un'altra cartella è che i worktree condividono lo stesso .git/ — stessa history, stessi remote, stesse credenziali. Non puoi avere lo stesso branch su due worktree contemporaneamente. È una feature relativamente di nicchia ma molto elegante quando serve.

9.1 — git cherry-pick: portare un commit dove vuoi

git cherry-pick prende uno o più commit specifici e li ri-applica su un altro branch, creando nuovi commit con lo stesso diff ma SHA diverso. Non sposta il commit originale — lo copia.

Il nome è evocativo: immagina un albero con tanti frutti (commit). Con cherry-pick scegli esattamente quale frutto raccogliere, uno alla volta, senza portarti dietro tutto il ramo.

Quando usarlo (e quando no)

Cherry-pick è lo strumento giusto in scenari precisi:

Hotfix già committato su un branch sbagliato. Hai fixato un bug su feature/login ma quel fix serve anche su main adesso, senza aspettare la merge della feature.

Backport su una versione precedente. Il tuo team mantiene v1.x in parallelo a v2.x. Un fix critico fatto su v2 deve essere portato anche su v1.

Recuperare un commit da un branch abbandonato. Un branch è stato chiuso senza merge, ma c'era un commit specifico che vale la pena salvare.

Invece non usare cherry-pick per portare intere feature da un branch all'altro — per quello esistono merge e rebase. Cherry-pick di molti commit in sequenza è quasi sempre un segnale che stai facendo la cosa sbagliata.

Il rischio delle dipendenze tra commit

Prima di eseguire qualsiasi cherry-pick, devi rispondere a una domanda: il commit che voglio copiare dipende da commit precedenti? Se sì, stai potenzialmente portando solo metà del lavoro — e Git non te lo impedisce.

Il problema si manifesta in due modi, con gravità molto diversa:

Caso 1 — conflitto immediato (il caso fortunato). Git prova ad applicare il diff e non riesce perché il codice su cui dovrebbe appoggiarsi non esiste sul branch di destinazione. Ti blocca con un conflitto. Almeno te ne accorgi subito.

Caso 2 — nessun conflitto, ma il codice è rotto (il caso pericoloso). Git applica il diff senza problemi perché sintatticamente non ci sono sovrapposizioni — ma il risultato non funziona. Esempio concreto:

# Commit A: aggiunge il metodo CalcolaSconto() in OrdineService.cs
# Commit B: chiama CalcolaSconto() da un nuovo controller

# Se cherry-picki solo B senza A:
# → Git non si lamenta (nessun conflitto di righe)
# → il codice compila
# → a runtime esplode perché CalcolaSconto() non esiste sul branch di destinazione
# → oppure esiste già una versione vecchia con firma diversa: comportamento silenziosamente sbagliato

La contromossa è semplice: prima di cherry-pickare, leggi la catena.

# Guarda i commit del branch sorgente
git log --oneline feature/login

# Per ogni commit che ti interessa, leggi il diff completo
git show a3f9c12

# Se B dipende da A, porta entrambi in ordine con la sintassi range
git cherry-pick A^..B

Se la catena di dipendenze è lunga o intrecciata, cherry-pick sta diventando lo strumento sbagliato — valuta un merge mirato o una PR dedicata.

💡 Il commit ideale per cherry-pick

Cherry-pick è sicuro quanto più il commit è autocontenuto: tocca un solo file o un perimetro preciso, risolve un bug specifico, non dipende da roba introdotta nello stesso branch. Un commit che fa parte di una feature sviluppata in 10 step dipendenti l'uno dall'altro è il candidato peggiore possibile.

⚠️ Cherry-pick crea duplicati nella history

Il commit originale rimane dov'era. Il cherry-pick crea un nuovo commit con lo stesso diff e messaggio, ma SHA diverso. Se in futuro fai merge del branch originale, Git potrebbe vedere quel diff come duplicato e gestirlo in modo strano. Usalo con criterio e documentati nei messaggi di commit.

Il comando base è semplice: ti posizioni sul branch di destinazione e cherry-picki lo SHA del commit che vuoi copiare.

9.3 — Cherry-pick di più commit e range

Puoi cherry-pickare più commit in un'unica operazione, sia elencandoli esplicitamente sia usando la sintassi a range.

Più commit singoli (non contigui)

# Cherry-pick di tre commit specifici, nell'ordine indicato
git cherry-pick a3f9c12 e7b2d90 f1c3a44

Git li applica nell'ordine in cui li scrivi, creando un commit separato per ognuno.

Range contiguo di commit

# Sintassi: A..B significa "tutti i commit dopo A fino a B incluso"
# ATTENZIONE: A è ESCLUSO, B è incluso
git cherry-pick a3f9c12..f1c3a44

# Se vuoi includere anche A, usa la sintassi con il cappello:
git cherry-pick a3f9c12^..f1c3a44
Branch feature/login:
A ── B ── C ── D ── E
git cherry-pick B..D → porta C e D (B escluso, D incluso)
git cherry-pick B^..D → porta B, C e D (B incluso)
Branch main dopo il cherry-pick:
X ── Y ── C' ── D' (nuovi SHA, stesso diff)

Flag --no-commit: cherry-pick senza committare subito

Con --no-commit (o -n) Git applica le modifiche allo staging area ma non crea il commit. Utile quando vuoi combinare le modifiche di più cherry-pick in un unico commit, o vuoi revisionare prima di committare.

# Applica i diff di tre commit in staging, senza committare
git cherry-pick -n a3f9c12 e7b2d90 f1c3a44

# Rivedi le modifiche
git diff --staged

# Poi committi tutto in uno solo
git commit -m "backport: fix sconto e validazione email (da feature/login)"

9.4 — Conflitti durante cherry-pick

Cherry-pick applica un diff su un contesto diverso da quello in cui era stato creato. Se nel frattempo il codice circostante è cambiato, Git non riesce ad applicare il diff in automatico e si ferma con un conflitto — esattamente come durante un merge.

git cherry-pick a3f9c12
# Output:
# error: could not apply a3f9c12... fix: corretto calcolo sconto
# hint: After resolving the conflicts, mark them with
# hint:   git add <pathspec> ... and then
# hint:   git cherry-pick --continue

# 1. Apri i file in conflitto, risolvi manualmente i marcatori <<<<<<< / >>>>>>>
# 2. Aggiungi i file risolti allo staging
git add src/OrdineService.java

# 3. Continua il cherry-pick
git cherry-pick --continue
# Git apre l'editor per il messaggio di commit — puoi modificarlo o lasciarlo

# OPPURE: se vuoi abbandonare tutto e tornare allo stato pre-cherry-pick
git cherry-pick --abort
ParteSignificato
git cherry-pick --continuedopo aver risolto i conflitti e fatto git add, prosegue e crea il commit
git cherry-pick --abortannulla l'intera operazione e ripristina lo stato pre-cherry-pick
git cherry-pick --skipin un range multi-commit, scarta il commit corrente in conflitto e passa al prossimo — usalo solo se sei sicuro che quel diff è già presente sul branch di destinazione in altra forma (vedi esempio sotto)

Esempio completo: --skip in azione

Stai portando tre commit da feature/login su main con un range:

git cherry-pick A^..C
# Git applica A → ok, commit creato
# Git applica B → CONFLITTO, si ferma

# A questo punto hai tre scelte:

# 1. Risolvi il conflitto e prosegui normalmente
git add src/FileConflittato.cs
git cherry-pick --continue   # applica B risolto, poi passa a C

# 2. Salti B perché quel diff è già presente su main in altra forma
git cherry-pick --skip       # scarta B completamente, applica C

# 3. Molli tutto
git cherry-pick --abort      # torna allo stato pre-cherry-pick, nessun commit creato
⚠️ --skip non è "ignoro il conflitto e speriamo bene"

Usalo solo quando sei certo che il diff del commit in conflitto è già presente sul branch di destinazione — ad esempio perché qualcuno lo ha già portato manualmente in precedenza. Se salti un commit da cui dipendono quelli successivi del range, stai reintroducendo esattamente il problema delle dipendenze tra commit discusso nella sezione 9.1.

ℹ️ Cherry-pick in conflitto su Azure DevOps

Azure DevOps non offre cherry-pick dalla UI web per i branch di PR in modo diretto. Il flusso consigliato è sempre da CLI: cherry-pick in locale → risolvi conflitti → push → apri PR sul branch di destinazione. Alcuni team usano cherry-pick proprio per preparare una PR di backport pulita verso un branch di release.

9.5 — Cherry-pick: best practice e insidie

Documenta sempre l'origine

Quando fai cherry-pick, è buona norma lasciare traccia dell'SHA originale nel messaggio di commit. Aiuta chiunque legga la history a capire da dove viene quel commit.

git cherry-pick -x a3f9c12
# Il flag -x aggiunge automaticamente in fondo al messaggio:
# "(cherry picked from commit a3f9c12...)"

Evita cherry-pick su commit di merge

Un merge commit ha due genitori, e Git quando deve calcolare il diff non sa rispetto a quale dei due farlo. Considera questa situazione:

main: A ── B ── M ← M è il merge commit di feature/login in main
/
feature/login: C ── D ──
M ha due genitori: B (su main) e D (su feature/login)
Il "diff" di M dipende da quale dei due usi come riferimento — e sono diff completamente diversi

Se provi a cherry-pickare M senza specificare nulla, Git si blocca immediatamente:

git cherry-pick M
# error: commit M is a merge but no -m option was given.
# fatal: cherry-pick failed

# Devi usare -m per indicare quale genitore usare come "mainline"

# -m 1 = primo genitore (B, quello su main)
# diff = "cosa ha portato feature/login rispetto a main" ← di solito quello che vuoi
git cherry-pick -m 1 M

# -m 2 = secondo genitore (D, quello su feature/login)
# diff invertito ← raramente quello che vuoi
git cherry-pick -m 2 M

Anche con -m 1 il risultato può essere inaspettato: stai chiedendo a Git di ricavare un diff da un commit che per sua natura rappresenta una fusione di due storie, non una modifica atomica. Il diff può essere enorme, parziale, o includere modifiche che non volevi portare.

La soluzione corretta è quasi sempre un'altra: invece di cherry-pickare il merge commit, identifichi i singoli commit normali dentro feature/login che ti interessano e cherry-picki quelli.

# ❌ Opaco e rischioso
git cherry-pick -m 1 M

# ✅ Preciso e controllato — cherry-pick dei commit atomici
git cherry-pick C^..D
⚠️ Cherry-pick su merge commit: last resort

Trattalo come ultima spiaggia, non come approccio standard. Prima chiediti sempre: posso identificare i singoli commit dentro il branch sorgente che mi interessano? Quasi sempre la risposta è sì — e quella è la strada giusta.

⚠️ Il problema dei duplicati nella history

Se cherry-picki il commit A da feature su main, otterrai A' su main. Quando poi fai merge di feature su main, Git vedrà A e A' come due commit distinti con lo stesso diff. Nella maggior parte dei casi Git è abbastanza intelligente da ignorare il duplicato, ma può creare confusione nella log history. Il flag -x aiuta proprio a documentare questa relazione.

⚠️ Non fare cherry-pick durante una sessione bisect attiva

Durante git bisect sei in detached HEAD — non sei su nessun branch. Se in quello stato esegui un cherry-pick, Git lo applica tecnicamente su HEAD ma il commit risultante non appartiene a nessun branch. Quando poi esegui git bisect reset, torni al branch di partenza e quel commit sembra sparito (è nel reflog, ma è un casino da recuperare). Regola semplice: finisci sempre la sessione bisect con git bisect reset prima di fare qualsiasi altra operazione. (→ Modulo 8: git bisect)

9.6 — git reflog avanzato: recuperare l'irrecuperabile

Il reflog è il registro interno di Git che tiene traccia di ogni movimento di HEAD — branch switch, commit, rebase, reset, cherry-pick. Tutto. È la tua rete di sicurezza assoluta: anche dopo un git reset --hard o un rebase andato storto, il reflog ti permette di recuperare commit che sembrano persi.

Abbiamo già visto il reflog nel Modulo 7 — sezione 7.5 in relazione a reset. Qui andiamo più in profondità.

Struttura del reflog

git reflog
# Output:
# e1b7f33 HEAD@{0}: cherry-pick: fix: corretto calcolo sconto
# a91c3d2 HEAD@{1}: checkout: moving from feature/login to main
# f7e2109 HEAD@{2}: commit: feat: aggiunta pagina profilo utente
# b3d8a11 HEAD@{3}: rebase (finish): returning to refs/heads/feature/login
# 9c41f7e HEAD@{4}: rebase (pick): refactor: pulizia metodi deprecati
# ...

Ogni riga ha il formato: <SHA> HEAD@{N}: <operazione>: <descrizione>. HEAD@{0} è sempre lo stato corrente, HEAD@{1} quello precedente, e così via.

Recuperare commit dopo un reset --hard

Scenario classico: hai fatto git reset --hard e hai perso commit che non avresti dovuto perdere.

# Hai fatto questo per sbaglio (o con troppa fretta)
git reset --hard HEAD~3
# Tre commit volatilizzati dalla branch history

# 1. Apri il reflog e trova lo SHA dell'ultimo commit che avevi PRIMA del reset
git reflog
# f7e2109 HEAD@{1}: commit: feat: aggiunta pagina profilo   ← questo era il tuo HEAD

# 2. Torna a quello stato
git reset --hard HEAD@{1}
# oppure usa lo SHA diretto
git reset --hard f7e2109

# I tre commit sono tornati. Nessun dato perso.

Recuperare commit dopo un rebase andato storto

# Il rebase ha creato un casino — vuoi tornare allo stato pre-rebase
git reflog
# b3d8a11 HEAD@{0}: rebase (finish): returning to refs/heads/feature/login
# 9c41f7e HEAD@{1}: rebase (pick): ...
# ...
# a72f8cd HEAD@{7}: checkout: moving from main to feature/login  ← PRIMA del rebase

# Torna allo stato pre-rebase con un reset hard
git reset --hard HEAD@{7}
# oppure cerca la voce "rebase (start)" nel reflog — è sempre lì

Recuperare un branch cancellato

# Hai cancellato un branch per sbaglio
git branch -D feature/pagamento-paypal
# Deleted branch feature/pagamento-paypal (was c8e3f91).

# Il messaggio di git branch -D ti dice già lo SHA dell'ultimo commit!
# Ma se non l'hai visto, il reflog ce l'ha comunque:
git reflog
# c8e3f91 HEAD@{3}: commit: feat: integrazione PayPal base

# Ricrea il branch da quello SHA
git branch feature/pagamento-paypal c8e3f91
# Branch recuperato integralmente

Reflog per branch specifici

# Il reflog generico mostra i movimenti di HEAD
# Puoi vedere il reflog di un branch specifico:
git reflog show main
git reflog show feature/login

# Utile per capire cosa è successo su un branch anche quando non eri su HEAD

Scadenza del reflog e pulizia

Il reflog non è eterno: di default Git conserva le voci per 90 giorni (30 giorni per le voci "unreachable", cioè commit non raggiungibili da nessun branch). Dopo, vengono cancellate dal garbage collector.

# Vedi la configurazione attuale di scadenza
git config gc.reflogExpire
# default: 90.days.ago

# Forzare scadenza manuale (raramente serve, ma esiste)
git reflog expire --expire=now --all
# ATTENZIONE: cancella tutto il reflog immediatamente — difficilmente recuperabile

# Pulizia garbage collector manuale
git gc --prune=now
⚠️ Il reflog è locale, non viene pushato

Il reflog esiste solo nella tua copia locale del repository. Se cloni un repo da zero, o se lavori su una macchina diversa, il reflog non c'è. Non fare affidamento sul reflog per recuperare dati su ambienti diversi dal tuo — e non aspettare settimane prima di usarlo per un recupero.

9.7 — Git Hooks: automatizza i controlli

I Git hooks sono script che Git esegue automaticamente in risposta a eventi specifici: prima di un commit, dopo un push, prima di un merge, ecc. Vivono nella cartella .git/hooks/ del tuo repository e possono essere scritti in qualsiasi linguaggio con un interprete disponibile (bash, Python, Node, ecc.).

L'idea è semplice ma potente: vuoi che Git blocchi automaticamente un commit se i test falliscono? O che verifichi che il messaggio di commit rispetti un formato prestabilito? I hooks sono la risposta.

Dove vivono gli hooks

ℹ️ Come Git esegue gli hook su Windows

Gli script hook sono scritti in bash (#!/bin/bash) e vengono eseguiti da Git Bash, che Git for Windows include automaticamente. Git triggera gli hook internamente tramite sh/bash, indipendentemente dalla shell che stai usando tu (PowerShell, cmd, terminale di IntelliJ) — quindi funzionano senza aprire Git Bash manualmente. L'unica regola: scrivi e modifica sempre gli script in bash, non riscriverli in PowerShell, altrimenti Git non saprebbe eseguirli.

Linux (bash):

# Elenca tutti gli hook disponibili
ls .git/hooks/
# applypatch-msg.sample    post-update.sample       pre-merge-commit.sample
# commit-msg.sample        pre-applypatch.sample    pre-push.sample
# fsmonitor-watchman.sample pre-commit.sample        pre-rebase.sample
# post-commit.sample       pre-receive.sample       update.sample

# Attiva un hook: rimuovi .sample e rendilo eseguibile
cp .git/hooks/pre-commit.sample .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit

Windows (PowerShell o cmd):

# Elenca tutti gli hook disponibili
# ls è alias di Get-ChildItem in PowerShell — funziona uguale
ls .git/hooks/
# oppure esplicito:
Get-ChildItem .git/hooks/

# Attiva un hook: copia il file senza estensione .sample
# (chmod non serve su Windows)
copy .git\hooks\pre-commit.sample .git\hooks\pre-commit
ℹ️ Gli hooks NON vengono committati nel repository

La cartella .git/ non è tracciata da Git — quindi gli hooks non vengono condivisi con il team tramite push/clone. Per condividere hooks con il team esistono soluzioni come Husky (per progetti Node) o la configurazione di core.hooksPath su una cartella committata nel repo.

9.8 — Hook client-side: pre-commit, commit-msg, pre-push

Gli hook client-side girano in locale sulla tua macchina. Sono i più comuni e utili per la qualità del codice quotidiana. Gli script qui sotto vanno creati in bash — Git li eseguirà tramite Git Bash su Windows senza intervento manuale da parte tua.

pre-commit: blocca prima di committare

Gira prima che Git crei il commit. Se lo script esce con codice diverso da 0, il commit viene bloccato. Perfetto per: eseguire linting, formattazione automatica, test veloci, bloccare file sensibili.

Esempio 1 — blocca file con "TODO: FIXME"

Linux:

#!/bin/bash
# .git/hooks/pre-commit

if git diff --cached --name-only | xargs grep -l "TODO: FIXME" 2>/dev/null; then
  echo "❌ ERRORE: trovati TODO: FIXME nei file staged. Rimuovili prima di committare."
  exit 1
fi
exit 0

Windows (Git Bash — stesso script, funziona identico):

#!/bin/bash
# .git/hooks/pre-commit
# Su Windows Git esegue questo script tramite Git Bash automaticamente

exit 0

Esempio 2 — esegui i test Java con Maven prima del commit

Linux:

fi

exit 0

commit-msg: valida il formato del messaggio

Gira dopo che hai scritto il messaggio di commit ma prima che il commit venga creato. Riceve come argomento il path del file temporaneo che contiene il messaggio. Se esce con codice diverso da 0, il commit è bloccato.

Linux:

#!/bin/bash
# .git/hooks/commit-msg
# Forza il formato Conventional Commits: tipo(scope): descrizione
# Esempi validi: feat: aggiunta login, fix(api): corretto timeout

MSG_FILE=$1
MSG=$(cat "$MSG_FILE")
PATTERN="^(feat|fix|docs|style|refactor|test|chore|perf|ci|build|revert)(\(.+\))?: .{1,72}"

if ! echo "$MSG" | grep -qE "$PATTERN"; then
  echo "Formato messaggio non valido."
  echo "Usa: tipo(scope): descrizione"
  echo "Esempi: feat: aggiunta login | fix(api): corretto timeout"
  echo "Tipi validi: feat fix docs style refactor test chore perf ci build revert"
  exit 1
fi
exit 0

Se salvi uno script hook su Windows con line ending CRLF (\r\n), Git Bash non riesce a interpretarlo e l'hook semplicemente non gira — senza errore visibile, il che lo rende difficile da diagnosticare. Salva sempre gli script hook con line ending LF (\n). In IntelliJ il formato è visibile in basso a destra nella barra di stato — clicca su "CRLF" e seleziona "LF". In alternativa: git config core.autocrlf false nel repository.

pre-push: blocca prima di pushare

Gira prima che Git invii i commit al remote. Ottimo per bloccare push diretti su branch protetti, o per eseguire una suite di test più completa.

  exit 1
fi

9.9 — Condividere gli hooks con il team

Poiché .git/hooks/ non è versionato, ogni membro del team deve configurarsi gli hook manualmente. Esistono due approcci principali per risolvere questo problema.

Approccio 1: cartella committata + core.hooksPath

È l'approccio nativo Git, senza dipendenze esterne. Crei una cartella nel repository (es. .githooks/), ci metti i tuoi script, e dici a Git di usare quella cartella invece di .git/hooks/.

Linux:

# 1. Crea la cartella nel repository
mkdir .githooks

# 2. Crea i tuoi script lì (pre-commit, commit-msg, ecc.) e rendili eseguibili
chmod +x .githooks/*

# 3. Aggiungi e committi la cartella
git add .githooks/
git commit -m "chore: aggiunta cartella hooks condivisi"

# 4. Ogni sviluppatore, dopo il clone, esegue:
git config core.hooksPath .githooks

Windows (PowerShell o cmd):

# 1. Crea la cartella nel repository

# 2. Crea i tuoi script lì (pre-commit, commit-msg, ecc.)
#    chmod non serve su Windows — Git Bash non controlla i permessi di esecuzione
#    MA ricorda: salva gli script con line ending LF, non CRLF

git config core.hooksPath .githooks
ℹ️ core.hooksPath su Windows: il path usa slash, non backslash

Anche su Windows, core.hooksPath vuole il path con slash (.githooks o .githooks/), non con backslash. Git lo gestisce internamente in formato Unix — scrivere .githooks\ può causare problemi. Usa sempre la forma con slash o senza separatore finale.

Approccio 2: Husky (progetti Node/npm)

Se il tuo progetto usa npm, Husky è lo standard de facto per la gestione degli hook. Configura automaticamente core.hooksPath dopo un npm install, quindi ogni sviluppatore che clona e installa le dipendenze ottiene gli hook senza fare niente di extra.

# Installazione Husky in un progetto Node
npm install husky --save-dev
npx husky init

# Aggiunge un pre-commit hook tramite Husky
echo "npm test" > .husky/pre-commit

# Da quel momento, ogni git commit eseguirà npm test automaticamente
# e bloccherà il commit se i test falliscono
💡 Per progetti .NET, usa core.hooksPath

Husky dipende da npm, che non è uno strumento tipico di un progetto .NET/C#. Per i tuoi progetti su Azure DevOps con .NET, l'approccio con .githooks/ e core.hooksPath è più pulito e senza dipendenze extra. Documentalo nel README del progetto come step di setup obbligatorio per i nuovi sviluppatori.

9.10 — Git Submodule: repository dentro repository

Un submodule è un repository Git annidato dentro un altro repository Git. Il repository "padre" tiene traccia non del contenuto del submodule, ma di uno specifico commit di quel submodule. È uno strumento potente ma famoso per essere fonte di confusione — usalo solo quando le alternative (package manager, monorepo) non sono praticabili.

Quando i submodule hanno senso

Il caso d'uso classico è avere una libreria condivisa che viene sviluppata autonomamente e inclusa come dipendenza in più progetti, ma vuoi che ogni progetto "pinni" una versione specifica della libreria (un commit preciso), non l'ultimo. Un package manager come NuGet copre molti di questi casi meglio dei submodule — usa i submodule quando hai bisogno di modificare il codice della dipendenza direttamente e committare su entrambi i repository.

Aggiungere un submodule

# Aggiunge il repository "libreria-comune" come submodule nella cartella libs/comune
git submodule add https://dev.azure.com/miorg/libreria-comune/_git/libreria-comune libs/comune

# Git crea due cose:
# 1. La cartella libs/comune con il contenuto del repo
# 2. Il file .gitmodules nella root del tuo progetto

cat .gitmodules
# [submodule "libs/comune"]
#     path = libs/comune
#     url = https://dev.azure.com/miorg/libreria-comune/_git/libreria-comune

# Committi la configurazione del submodule
git commit -m "chore: aggiunto submodule libreria-comune"

Clonare un repository con submodule

# Opzione 1: clone normale + init + update (tre step)
git clone <url-progetto-padre>
git submodule init
git submodule update

# Opzione 2: tutto in uno (preferibile)
git clone --recurse-submodules <url-progetto-padre>
⚠️ Clone senza --recurse-submodules = cartelle vuote

Se fai git clone senza --recurse-submodules su un repository con submodule, le cartelle dei submodule esistono ma sono vuote. È uno degli errori più comuni con i submodule. Ricordati sempre del flag, oppure usa git submodule update --init --recursive dopo un clone normale.

Aggiornare un submodule all'ultimo commit

# Il progetto padre punta a un commit specifico del submodule.
# Per aggiornarlo all'ultimo commit del branch main del submodule:
cd libs/comune
git pull origin main
cd ../..

# Ora il progetto padre vede il submodule come "modificato" (nuovo commit puntato)
# Devi committare questa modifica nel progetto padre
git add libs/comune
git commit -m "chore: aggiornato submodule libreria-comune a commit abc1234"

# Oppure puoi aggiornare tutti i submodule in un colpo solo:
git submodule update --remote --merge

Amend Riscrittura dell'ultimo commit locale: cambia messaggio e/o contenuto producendo un nuovo SHA. Non usare su commit già pushati. Mod 8 §8.3
Bisect Ricerca binaria automatica del commit che ha introdotto un bug. Git dimezza ad ogni passo il range di commit da ispezionare. Mod 8 §8.5
Blob Oggetto interno Git che contiene il contenuto grezzo di un file. Non ha nome né path — è solo dati, identificato dal suo hash SHA-1. Mod 1 §Oggetti
Branch Puntatore mobile a un commit. Creare un branch non copia nulla — è solo un file di testo con un hash SHA-1 dentro. Mod 2 · Mod 3
Cherry-pick Copia un singolo commit (o range) su un branch diverso, creando un nuovo commit con lo stesso diff ma SHA differente. L'originale rimane dov'era. Mod 9 §9.1
Commit Snapshot immutabile dello stato del progetto. Contiene puntatore al tree root, hash del commit padre, autore, timestamp e messaggio. Mod 1
Conflict marker Marcatori <<<<<<< / ======= / >>>>>>> che Git inserisce nei file quando non riesce a risolvere automaticamente un conflitto durante merge, rebase o cherry-pick. Mod 5 §5.3
DAG Directed Acyclic Graph — la struttura dati che Git usa per la history. Ogni commit punta ai suoi genitori, mai in avanti. Non esistono cicli. Mod 1
Detached HEAD Stato in cui HEAD punta direttamente a un commit invece che a un branch. I nuovi commit non appartengono a nessun branch e sembrano "sparire" al checkout successivo. Mod 3 §3.3
Divergenza Situazione in cui il branch locale e il suo tracking remoto hanno entrambi commit esclusivi. Un semplice push viene rifiutato — serve fetch + merge o rebase prima. Mod 4 §4.4
Fast-forward Tipo di merge in cui il branch di destinazione non ha commit propri rispetto al sorgente. Git sposta semplicemente il puntatore in avanti senza creare un merge commit. Mod 2 · Mod 5 §5.1
Fetch Scarica dal remote i nuovi oggetti e aggiorna i tracking branch remoti (es. origin/main). Non tocca il working tree né i branch locali. Mod 4 §4.1
Force push Push che sovrascrive la history remota. Necessario dopo un rebase. Usare sempre --force-with-lease invece di --force per evitare di sovrascrivere commit di altri. Mod 6
HEAD Puntatore che indica su quale branch (o commit) stai lavorando ora. Di solito punta a un branch, che a sua volta punta all'ultimo commit. Mod 3
Hook Script bash eseguito automaticamente da Git in risposta a eventi (pre-commit, commit-msg, pre-push, ecc.). Vive in .git/hooks/ e non viene versionato. Mod 9 §9.7
Merge base Il commit antenato comune più recente tra due branch. Git lo usa come punto di riferimento nell'algoritmo 3-way merge per capire "chi ha cambiato cosa". Mod 5 §5.2
Merge commit Commit con due genitori creato da un merge non fast-forward. Rappresenta esplicitamente la convergenza di due linee di sviluppo nel grafo. Mod 5 §5.1
Object store Il database interno di Git in .git/objects/. Contiene tutti i blob, tree, commit e tag del repository. Gli oggetti sono immutabili e identificati da hash SHA-1. Mod 1
--ours / --theirs Flag per risolvere conflitti privilegiando una versione. Nel merge: ours = branch corrente, theirs = branch in ingresso. Nel rebase la logica si inverte. Mod 5 §5.4 · Mod 6
Pull Combinazione di fetch + merge (o fetch + rebase con --rebase). Aggiorna il branch locale con i nuovi commit del remote. Mod 4 §4.1
Pull Request (PR) Richiesta formale di integrare un branch in un altro, gestita dalla piattaforma (Azure DevOps). Include revisione del codice, discussioni e approvazioni prima del merge. Mod 2 · Mod 4
Rebase Riscrive i commit di un branch applicandoli sopra un altro punto di partenza. Produce una history lineare ma cambia gli SHA — non usare su commit già condivisi. Mod 6
Rebase interattivo git rebase -i: permette di riordinare, unire (squash), modificare o eliminare commit prima di pubblicarli. Strumento fondamentale per tenere la history pulita. Mod 6 §6.4
Reflog Registro locale di tutti i movimenti di HEAD. Permette di recuperare commit "persi" dopo reset --hard, rebase sbagliati o branch cancellati. Scade dopo 90 giorni. Mod 7 §7.5 · Mod 9 §9.6
Remote Versione del repository ospitata su un server (es. Azure DevOps). Di solito chiamato origin. È semplicemente un'altra copia del repo usata come punto di sincronizzazione. Mod 4
Reset Sposta HEAD (e il branch corrente) a un commit precedente. Tre modalità: --soft (tieni staging), --mixed (tieni file), --hard (butta tutto). Mod 7 §7.3
Restore Ripristina file nel working tree o nello staging area a una versione precedente, senza toccare HEAD. Non riscrive la history. Mod 7 §7.2
Revert Crea un nuovo commit che annulla le modifiche di un commit precedente. Non riscrive la history — è il modo sicuro per annullare su branch condivisi. Mod 7 §7.4
SHA-1 / hash Identificatore univoco a 40 caratteri esadecimali di ogni oggetto Git. È calcolato dal contenuto — oggetti identici hanno sempre lo stesso hash. Mod 1
Snapshot Fotografia dello stato completo di tutti i file del progetto in un dato momento. Ogni commit in Git è uno snapshot, non una lista di differenze. Mod 1
Squash merge Tipo di merge che comprime tutti i commit del branch sorgente in un unico commit sul branch di destinazione. Storia pulita, ma il cherry-pick futuro dei singoli commit non è più possibile. Mod 5 §5.5 · Mod 6
Staging area (Index) Area intermedia tra working tree e repository. Le modifiche aggiunte con git add vanno qui prima di diventare un commit. Permette di scegliere con precisione cosa committare. Mod 1
Stash Stack temporaneo dove salvare modifiche non committate per pulire il working tree senza perderle. Utile per cambiare branch in emergenza. Mod 8 §8.1
Submodule Repository Git annidato dentro un altro repository. Il repo padre tiene traccia di uno specifico commit del submodule, non del suo contenuto. Mod 9 §9.10
Tag Etichetta permanente su un commit specifico. I tag annotati (consigliati) includono messaggio, autore e data. Non si spostano come i branch. Mod 8 §8.2
3-way merge Algoritmo di merge che usa tre versioni: il file nella merge base, la versione su HEAD e la versione sul branch in ingresso. Confronta le differenze rispetto all'antenato comune. Mod 5 §5.2
Tracking branch Relazione tra un branch locale e il suo corrispettivo remoto (es. main → origin/main). Permette a Git di sapere se sei avanti, dietro o divergente rispetto al remote. Mod 4 §4.3
Tree Oggetto interno Git che rappresenta una directory. Contiene riferimenti a blob (file) e altri tree (sottodirectory), identificati da nome e hash. Mod 1
Worktree Working tree aggiuntivo collegato allo stesso repository. Permette di lavorare su branch diversi in cartelle separate contemporaneamente, senza clonare e senza stash. Mod 8 §8.6
